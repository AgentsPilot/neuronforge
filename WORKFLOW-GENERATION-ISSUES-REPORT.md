# Workflow Generation Issues Report

**Audience**: Workflow Generator Developers
**Date**: 2026-03-22
**Workflow**: Invoice Extraction (Agent ID: `43ffbc8a-406d-4a43-9f3f-4e7554160eda`)

---

## Executive Summary

During testing of the Invoice Extraction workflow, we identified **two critical issues** that affect workflows generated by the system. Both issues stem from the **workflow generation layer** creating configs that don't align with how the execution layer processes parameters.

**Impact**: These issues cause runtime failures that are difficult to debug, as the generated workflows appear valid but fail during execution.

**Status**: Both issues have been fixed at the **plugin executor level** as workarounds, but the root cause should be addressed in the **workflow generation system**.

---

## Issue #1: Variable Resolution Produces Nested Objects Instead of Scalar Values

### Problem Description

When a workflow step returns an object (e.g., file metadata) and a subsequent step needs just one field from that object (e.g., `file_id`), the workflow generator creates a config that references the entire object variable instead of extracting the specific field.

### Example from Invoice Extraction Workflow

**Step 8 (upload_file)** returns:
```json
{
  "file_id": "1abc123xyz",
  "file_name": "invoice.pdf",
  "file_size": "100 KB",
  "mime_type": "application/pdf",
  "web_view_link": "https://...",
  "uploaded_at": "2024-01-15T12:30:00Z"
}
```

**Step 9 (share_file)** config generated by workflow system:
```json
{
  "file_id": "{{drive_file}}",
  "permission_type": "anyone_with_link",
  "role": "reader"
}
```

### What Happens at Runtime

1. Variable resolver resolves `{{drive_file}}` to the **entire object**
2. Variable resolver **stringifies the object to JSON** before passing to plugin
3. Plugin receives: `file_id: "{ \"file_id\": \"1abc123xyz\", ... }"`
4. Google Drive API receives the JSON string in the URL → **404 Not Found**

### Expected Behavior

The workflow generator should either:
- **Option A**: Generate `"file_id": "{{drive_file.file_id}}"` (extract field in config)
- **Option B**: Teach variable resolver to auto-extract matching fields
- **Option C**: Document that plugins should handle this (current workaround)

### Root Cause Analysis

**Workflow Generation Layer**:
- The LLM-based workflow generator sees step 8 returns a "file" object
- It references the variable as `{{drive_file}}` assuming it represents the file_id
- No field extraction (`.file_id`) is added to the variable reference

**Why This Happens**:
- The generator may not have clear guidance on when to use object references vs. field extraction
- The plugin schema doesn't explicitly indicate which fields are "primary identifiers"
- The variable resolution system doesn't know the parameter type expectations

### Fix Applied (Workaround)

**Location**: `/lib/server/base-plugin-executor.ts` (lines 207-249)

Added `unwrapParameter()` method that:
1. Detects when a parameter receives a JSON string
2. Parses the JSON to an object
3. Checks if the object has a field matching the parameter name
4. Extracts and returns just that field value

**Example**:
```typescript
// Parameter receives: '{"file_id":"123","file_name":"doc.pdf",...}'
// unwrapParameter(value, 'file_id') returns: "123"
```

**Scope**: Applied to all Google Drive plugin parameters (`file_id`, `folder_id`, `parent_folder_id`)

### Why This is Not a Complete Solution

❌ **Only fixes Google Drive plugin** - Other plugins (Gmail, Sheets, etc.) need the same fix
❌ **Runtime overhead** - JSON parsing on every parameter
❌ **Masks the real problem** - Workflow generator should create correct configs
❌ **Hard to debug** - Developers won't see the unwrapping in logs unless they look closely
❌ **No validation** - If field doesn't exist, falls back to passing the whole object

### Recommended Fix for Workflow Generator

1. **Add field extraction logic** when generating variable references:
   ```typescript
   // Current (wrong)
   config.file_id = "{{drive_file}}"

   // Correct
   config.file_id = "{{drive_file.file_id}}"
   ```

2. **Enhance plugin schemas** with `x-primary-field` hint:
   ```json
   "output_schema": {
     "properties": {
       "file_id": {
         "type": "string",
         "x-primary-field": true,  // ← NEW
         "x-guaranteed": true
       }
     }
   }
   ```

3. **Update LLM prompts** to include explicit guidance:
   ```
   When a parameter expects a scalar value (string, number) but the variable
   holds an object, extract the matching field using dot notation:

   ✅ CORRECT: "file_id": "{{drive_file.file_id}}"
   ❌ WRONG:   "file_id": "{{drive_file}}"
   ```

4. **Add validation** at workflow generation time:
   - Check if parameter type is scalar but variable reference points to object
   - Warn or auto-correct by adding field extraction

## Issue #2: Schema Values Don't Match External API Requirements

### Problem Description

The plugin schema defines user-friendly parameter values (e.g., `"anyone_with_link"`), but the external API (Google Drive) expects different technical values (e.g., `"anyone"`). The workflow generator uses the schema values, but the plugin executor passes them directly to the API without normalization.

### Example from Invoice Extraction Workflow

**Plugin Schema** (`google-drive-plugin-v2.json` lines 1380-1389):
```json
"permission_type": {
  "type": "string",
  "enum": [
    "anyone_with_link",    // ← User-friendly
    "anyone_can_view",
    "anyone_can_edit",
    "specific_users"
  ]
}
```

**Google Drive API** expects:
```json
{
  "type": "anyone",        // ← Technical value
  "role": "reader"
}
```

**Step 9 config** generated by workflow system:
```json
{
  "file_id": "{{drive_file}}",
  "permission_type": "anyone_with_link",  // ← Schema value
  "role": "reader"
}
```

### What Happens at Runtime

1. Plugin executor receives: `permission_type: "anyone_with_link"`
2. Plugin executor passes it directly to Google API: `{ type: "anyone_with_link", role: "reader" }`
3. Google Drive API rejects with: **"Invalid value for: anyone_with_link is not a valid value"**

### Expected Behavior

Either:
- **Schema values should match API values** (less user-friendly)
- **Plugin executor should normalize values** (current workaround)
- **Workflow generator should know the mapping** and generate API values

### Root Cause Analysis

**Plugin Schema Design**:
- Prioritizes user experience with descriptive enum values
- Doesn't document the mapping to API values
- No indication that normalization is needed

**Workflow Generator**:
- Uses schema enum values as-is (correct behavior based on schema)
- Assumes plugin executor will handle the values (reasonable assumption)

**Plugin Executor**:
- Originally passed values directly to API (breaks with user-friendly values)
- No normalization layer between schema and API

### Fix Applied (Workaround)

**Location**: `/lib/server/google-drive-plugin-executor.ts` (lines 738-766)

Added normalization logic in `shareFile()`:
```typescript
const userPermissionType = parameters.permission_type || 'anyone_with_link';

if (userPermissionType === 'anyone_with_link' || userPermissionType === 'anyone_can_view') {
  permissionType = 'anyone';
  role = 'reader';
} else if (userPermissionType === 'anyone_can_edit') {
  permissionType = 'anyone';
  role = 'writer';
} else if (userPermissionType === 'specific_users') {
  permissionType = 'user';
  role = parameters.role || 'reader';
}
```

### Why This is Not a Complete Solution

❌ **Plugin-specific logic** - Every plugin needs custom normalization
❌ **Schema doesn't document mapping** - Developers must reverse-engineer API docs
❌ **Scattered normalization** - Some in executor, some in schema, hard to maintain
❌ **No validation** - If new enum value added to schema, executor might not handle it

### Recommended Fix for Plugin Schema Design

**Option A: Add API Value Mapping to Schema**
```json
"permission_type": {
  "type": "string",
  "enum": ["anyone_with_link", "anyone_can_view", "anyone_can_edit"],
  "x-api-mapping": {
    "anyone_with_link": { "type": "anyone", "role": "reader" },
    "anyone_can_view": { "type": "anyone", "role": "reader" },
    "anyone_can_edit": { "type": "anyone", "role": "writer" }
  }
}
```

**Option B: Use API Values Directly in Schema**
```json
"permission_type": {
  "type": "string",
  "enum": ["anyone", "user", "group", "domain"],
  "description": "Share with anyone (public link), specific users, groups, or domain"
}
```
- **Pros**: No normalization needed, values pass through directly
- **Cons**: Less user-friendly, requires users to understand API terminology

**Option C: Create Normalization Registry**
```typescript
// In base plugin executor or shared utility
const GOOGLE_DRIVE_MAPPINGS = {
  permission_type: {
    'anyone_with_link': { type: 'anyone', role: 'reader' },
    'anyone_can_view': { type: 'anyone', role: 'reader' },
    'anyone_can_edit': { type: 'anyone', role: 'writer' },
    'specific_users': { type: 'user' }
  }
};
```

## Technical Details for Implementation

### Field Extraction Logic Pseudocode

```typescript
function generateParameterValue(
  parameterSchema: ParameterSchema,
  variableReference: VariableRef,
  variableType: VariableType
): string {
  // If parameter expects scalar but variable is object
  if (isScalarType(parameterSchema.type) && isObjectType(variableType)) {

    // Check if object has field matching parameter name
    const paramName = parameterSchema.name;
    if (variableType.properties[paramName]) {
      // Use dot notation to extract field
      return `{{${variableReference.name}.${paramName}}}`;
    }

    // Check if there's a primary field hint
    const primaryField = findPrimaryField(variableType);
    if (primaryField) {
      return `{{${variableReference.name}.${primaryField}}}`;
    }

    // Warn: Cannot safely extract field
    console.warn(`Parameter ${paramName} expects scalar but variable is object`);
    return `{{${variableReference.name}}}`; // Let runtime unwrapping handle it
  }

  // Direct reference
  return `{{${variableReference.name}}}`;
}
```

### Schema Enhancement Example

```json
{
  "output_schema": {
    "type": "object",
    "properties": {
      "file_id": {
        "type": "string",
        "description": "ID of the uploaded file",
        "x-guaranteed": true,
        "x-primary-field": true,          // ← NEW: Hint for primary identifier
        "x-use-for-reference": true       // ← NEW: Use this field when referencing object
      },
      "file_name": {
        "type": "string",
        "description": "Name of the uploaded file",
        "x-guaranteed": true
      }
    },
    "x-reference-field": "file_id"       // ← NEW: Alternative, object-level hint
  }
}
```
