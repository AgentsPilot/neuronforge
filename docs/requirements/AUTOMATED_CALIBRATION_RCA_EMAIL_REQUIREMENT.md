# Requirement: Automated Calibration-Failure RCA in the Admin Alert Email

> **Last Updated**: 2026-07-05

**Created by:** BA
**Date:** 2026-07-05
**Status:** Approved (user sign-off 2026-07-05 — all open questions resolved)

## Overview

Today, when a background calibration run lands on `failed` / `needs_review`, the platform emails the system admins a **deterministic** failure alert (`lib/calibration/calibrationAdminAlert.ts`, IMP-2). That email lists the raw remaining issues and tells admins to go run `scripts/dump-calibration.ts <agentId>` and perform the root-cause analysis (RCA) **by hand**. This requirement adds a new production, runtime, LLM-driven capability: the system **automatically produces the RCA** (the same analysis the `calibration-rca` skill / Troubleshooter would produce) and delivers a **finished, structured root-cause analysis inside that same admin email**, so admins receive a hand-off-ready conclusion they can pass to the Team Leader (TL) to trigger a fix — instead of doing the RCA themselves.

The RCA is generated by a new **in-app RCA service** that reads the same persisted evidence the manual RCA reads. The manual method uses **two dumps** — `dump-calibration.ts` (the calibration outcome) **and** `dump-agent.ts` (the agent's workflow / compiled DSL) — so the service reads **both** the calibration outcome tables and the failed agent's workflow definition (via repositories), giving the runtime RCA parity with the human two-dump method, and calls the LLM through the **provider factory**. It is **best-effort with graceful fallback**: if RCA generation fails or exceeds a bounded timeout, the admin still receives today's deterministic alert. This **productionizes the RCA methodology**; it is NOT an edit to the Troubleshooter (TS) Claude-Code agent (which cannot run inside the Vercel serverless app).

## Problem & Motivation

The IMP-2 admin alert (`sendCalibrationAdminAlert`) is fast and deterministic by design: no LLM, system email transport, deduped by `workflow_hash`, internal-only (it embeds the customer input data the agent was processing). It does its job — it pages the admins — but it stops at *raw evidence*: the remaining issues array plus a `npx tsx scripts/dump-calibration.ts <agentId>` command and a pointer to `CALIBRATION_RCA_RUNBOOK.md`. Every failure therefore still requires an admin to manually pull the evidence bundle, walk the 6-step method, find the earliest failing step, classify the root-cause layer, and write up a conclusion before anyone can decide on a fix.

That manual RCA is now a well-defined, repeatable method (the `calibration-rca` skill + runbook) with a standardized conclusion shape (the Troubleshooter's 8-field conclusion + 5-value layer set). The method pulls evidence from **two** persisted sources: the calibration outcome (`calibration_sessions`, `calibration_history`, `agent_executions` — dumped by `dump-calibration.ts`) **and** the failed agent's workflow definition (the compiled DSL `pilot_steps`, `input_schema`, and Enhanced Prompt on the `agents` row — dumped by `dump-agent.ts`). `dump-calibration.ts` itself ends by instructing the investigator to open the failing step in `pilot_steps` (`dump-agent.ts <id>`) and classify the layer — so the workflow definition is not optional evidence, it is what makes the layer classification defensible. Because the method is repeatable and all of this evidence is already persisted, the RCA can be produced **automatically at failure time** and delivered in the alert — turning a "here's the raw failure, go investigate" page into a "here's the finished root cause + named fix-owner + recommended remediation path, route it" page. This shortens time-to-fix and makes the alert directly actionable by the TL.

The hard constraint: the current alert is intentionally fast and never-fails. An LLM RCA call is slower and *can* fail. So the RCA must be **additive and best-effort** — it augments the alert when it succeeds within budget, and silently falls back to today's deterministic content when it doesn't. The admin must never lose the alert because the RCA step failed.

## Actors / Personas

| Actor | Role in this feature |
|-------|----------------------|
| **System Admin** (recipient) | Receives the augmented alert. Reads the auto-generated RCA and hands it to the TL. Resolved via `AdminAccessService.listAdminEmails()` (as today). |
| **Team Leader (TL)** | Receives the RCA from the admin (or via the persisted record) and routes it — hotfix → SA→Dev, or larger → BA — exactly as in the Troubleshooter requirement. |
| **Background calibration run** (trigger) | The `POST /api/v2/calibrate/batch` background path that lands on `failed`/`needs_review` and, in its `finally` tail, fires the admin alert. This is the sole v1 trigger surface. |
| **In-app RCA service** (new) | Reads persisted calibration evidence **and the failed agent's workflow definition** via repositories, applies the `calibration-rca` method, calls the LLM through the provider factory, returns a Zod-validated structured RCA. |
| **Agent owner** | Not a recipient of this alert (internal-only). Their input data appears in the RCA prompt/evidence — see the privacy NFR. |

## User Stories

- As a **system admin**, I want the failure alert to already contain a finished root-cause analysis (earliest failing step, why, which layer owns it, a named fix-owner, and suggested solutions), so that I can hand it straight to the TL without pulling the evidence bundle and doing the RCA myself.
- As a **system admin**, I want to still receive the alert with the raw issues and the `dump-calibration.ts` guidance even when the automatic RCA can't be produced, so that a slow or failed RCA never costs me the alert.
- As a **Team Leader**, I want the auto-RCA to use the same conclusion shape as the Troubleshooter (8 fields + one of the 5 layers), so that I can route it with the same decision process I already use for TS conclusions.
- As a **Team Leader**, I want the auto-RCA to reason over the failing step's actual definition (the compiled DSL, input schema, and Enhanced Prompt), so that its layer classification is as defensible as a human RCA — e.g. distinguishing a V6-generation defect from an input/data problem — instead of only knowing which step failed.
- As a **platform owner**, I want the generated RCA persisted on a retrievable record keyed to the calibration, so that it can be linked/handed to the TL later and isn't lost when the email is archived.
- As a **platform owner**, I want the request `correlationId` persisted on the calibration record at failure time, so that a future log-drain integration can join platform logs to this failure without re-plumbing anything.
- As a **platform owner**, I want the RCA augmentation behind a feature flag and bounded by the serverless time budget, so that I can roll it out safely and it never delays or fails calibration completion.
- As a **platform owner**, I want the LLM RCA to respect the existing `workflow_hash` dedup, so that a broken agent version only incurs one RCA generation (bounded LLM cost/volume).

## Functional Requirements

### A. Trigger scope & placement

| # | Requirement |
|---|-------------|
| **FR-1** | The auto-RCA MUST be produced **only** on the existing calibration admin-alert path: the background (`isBackground === true`) calibration tail in `app/api/v2/calibrate/batch/route.ts` `finally` block, for runs that did not pass (`!passed`) and have a `workflow_hash`, immediately before/around the existing `sendCalibrationAdminAlert(...)` call. No other lifecycle stage (runtime executions, agent creation, V6 generation) is a trigger in v1 (see Out of Scope). |
| **FR-2** | The auto-RCA MUST respect the existing dedup: it MUST only run for a `(agentId, userId, workflowHash)` that has NOT already been alerted (`hasAdminAlertBeenSent` returns false). When the alert is skipped as a duplicate, the RCA MUST NOT be generated (one RCA per broken agent version — bounds LLM cost/volume). |

### B. Delivery — augment the single existing alert email

| # | Requirement |
|---|-------------|
| **FR-3** | The generated RCA MUST be rendered as a **new section within the existing `calibrationAdminAlert` email** (not a separate email). There MUST remain exactly one admin alert email per broken agent version. |
| **FR-4** | The RCA section MUST be rendered in the same visual style as `renderAdminAlertHtml` (same HTML shell, section headings, `esc()` escaping of all dynamic content, dark code/`<pre>` blocks for technical detail). All RCA content MUST be HTML-escaped — no injection from issue text, input values, or LLM output. |
| **FR-5** | The email MUST retain **all** of today's deterministic content (agent & run table, remaining issues, auto-adjustments tried, "Data the agent was processing", the `dump-calibration.ts` command, runbook pointer, agent/sandbox links, admin-test banner, internal-only footer). The RCA is **additive** — it augments, never replaces, the existing content. |
| **FR-6** | When an RCA was successfully generated, the email SHOULD label the RCA section as auto-generated (e.g. "Automated RCA (LLM-generated — verify before acting)") so admins treat it as a machine-produced starting point, not a hand-verified conclusion. |

### C. Graceful fallback (best-effort, never lose the alert)

| # | Requirement |
|---|-------------|
| **FR-7** | RCA generation MUST be **best-effort with graceful fallback**. If RCA generation throws, returns invalid output (fails Zod validation), or exceeds a bounded timeout, the alert MUST STILL be sent with today's deterministic-only content. The admin MUST NEVER lose the alert because the RCA step failed. |
| **FR-8** | RCA generation MUST be bounded by an explicit timeout. The timeout is a **configurable value with a sensible code default**, resolved from the same DB-backed calibration-RCA config as the model (FR-13) — the `system_settings_config` entry, cached, with a code-level default fallback — and MUST respect the route's Vercel serverless ceiling (`export const maxDuration = 60` in the batch route), leaving headroom for the calibration work that precedes the `finally` tail and for the email send itself. If the timeout elapses, the code MUST proceed to send the deterministic alert (FR-7). |
| **FR-9** | Neither RCA generation nor the augmented send may throw out of the calibration tail. The overall alert path MUST preserve the existing `sendCalibrationAdminAlert` "never throws / returns boolean" contract — mirroring the existing best-effort, non-blocking tail. RCA generation MUST NOT fail or delay calibration completion, the route response, or the gate update. |
| **FR-10** | When RCA generation fails or times out, the code MUST log the failure via Pino (`logger.error({ err, agentId }, ...)` / `logger.warn`) with correlation context, and continue. Failure MUST NOT set the dedup marker in a way that blocks a later run from retrying (dedup marker semantics for the alert are unchanged — see FR-19). |

### D. The RCA service (in-app, provider-factory, repository-backed)

| # | Requirement |
|---|-------------|
| **FR-11** | A new in-app RCA service MUST be created (proposed: `lib/calibration/calibrationRcaService.ts`, exact location/name an SA/Dev detail) that, given the failed calibration identifiers (agentId, userId, sessionId, calibrationHistoryId, workflowHash, and the input values), produces a structured RCA by applying the `calibration-rca` methodology. |
| **FR-12** | The service MUST read the **same persisted evidence the manual RCA reads**, mirroring the human **two-dump** method (`dump-calibration.ts` + `dump-agent.ts`), in **two parts** — both via the repository layer, NOT by shelling out to `scripts/dump-calibration.ts` / `scripts/dump-agent.ts` and NOT via direct Supabase queries in the service (the dump scripts are CLI conveniences; production reads through repositories): **(a) calibration outcome evidence** — `calibration_sessions`, `calibration_history`, `agent_executions` — via `CalibrationHistoryRepository`, the calibration session repository, and any execution repository; **(b) the failed agent's workflow definition** — the compiled DSL steps (`pilot_steps`), the `input_schema`, and the Enhanced Prompt / `user_prompt` (plus `agent_config.ai_context` where relevant) — via **`AgentRepository.findById(agentId, userId)`** (owner-scoped read that returns the full `Agent` row, excluding deleted). Reading the workflow definition gives the runtime RCA **parity with the manual two-dump method**: without it the RCA would know *which* step failed but not the step's definition, weakening the "why" and, critically, the **root-cause layer classification** (esp. telling a **V6-generation** defect apart from an **input/data** problem). `dump-calibration.ts` itself ends by directing the investigator to open the failing step (`dump-agent.ts <id>`) to classify the layer — so the workflow definition is required evidence, not optional. |
| **FR-13** | The service MUST call the LLM **through the provider factory** (`getProviderFactory()` → `getProvider(...)` → `provider.complete(...)`). The **provider, model, and temperature MUST be resolved from a DB-backed config-with-defaults mechanism** — mirroring `lib/agentkit/v6/config/AgentGenerationConfigService.ts`: a dedicated **calibration-RCA config entry** in the **`system_settings_config`** table, read through a small cached accessor (~5-min in-memory TTL) with a hardcoded code-level `DEFAULT_CONFIG` fallback when the DB is unavailable. The service MUST NOT hardcode a model-name literal, MUST NOT use a static `factory.getDefaultModel(...)` or an env value for model selection, and MUST NOT call any provider SDK directly. Proposed code-default fallback: a **mid-tier reasoning model** (e.g. `{ provider: 'anthropic', model: <mid-tier Claude model>, temperature: 0 }`), tunable via DB config without a code change. (Rationale: one existing shadow validator, `ConstrainedSemanticValidator`, hardcodes its model — the RCA service MUST follow the good DB-config pattern, not that shortcut.) |
| **FR-14** | The service MUST apply the `calibration-rca` 6-step method as its prompt/reasoning contract: (1) evidence gathered (calibration outcome **and** workflow definition per FR-12), (2) read the issues array (separate the `steps_failed` aggregate from the specific per-step issues), (3) find the **earliest failing step** and trace the cascade — cross-referencing that step against its definition in `pilot_steps`, (4) classify the **root-cause layer**, (5) judge whether calibration itself behaved correctly, (6) conclude. The method MUST be sourced from the shared methodology, not re-invented (no plugin-specific hardcoding in the prompt — reference plugin schemas / evidence as the source of truth per Platform Design Principles). |
| **FR-15** | The LLM MUST return a **structured object** that is validated with a **Zod schema** at the service boundary before use. The schema MUST match the Troubleshooter conclusion shape — the **8 fields**: (1) reported **symptom**; (2) **evidence** gathered; (3) **earliest failing step + cascade**; (4) classified **root-cause layer**; (5) defensible **root cause** (the "why", with references); (6) named **fix-owner**; (7) one or more **suggested solutions**; (8) recommended **remediation path** (hotfix vs full cycle) — and the **root-cause layer** MUST be exactly one of the 5 TS values: **input/data · V6 generation · runtime/external API · calibration-detection · creation chat flow**. (The `calibration-rca` skill's 4-layer set is the calibration-scoped subset; the persisted+emailed RCA uses the TS 5-value set so it is routable identically to a TS conclusion.) |
| **FR-16** | The single validated RCA object MUST be usable for BOTH outputs: (a) persistence (FR-17) and (b) rendering into the email HTML section (FR-3/FR-4). If Zod validation fails, the service MUST return a typed "no RCA" result and the caller MUST fall back to deterministic-only (FR-7). |

### E. Persist a retrievable RCA record

| # | Requirement |
|---|-------------|
| **FR-17** | The generated RCA (the structured 8-field + 5-value-layer object) MUST be **persisted** to a retrievable record so it can be linked/handed to the TL later. **Target decided (user, 2026-07-05):** the `calibration_history` row for the failed run, via `CalibrationHistoryRepository`, stored under a new `metadata.auto_rca` key (JSON), alongside a `metadata.auto_rca_generated_at` timestamp and the model/provider used — **no migration**. Persistence MUST go through the repository (no direct Supabase write in the service/route). A dedicated column/table is revisited only if a future dashboard needs to query RCAs at scale (roadmap — see Out of Scope item 5). |
| **FR-18** | Persistence MUST be best-effort and MUST NOT block or fail the email send or calibration completion. If persistence fails, the RCA MUST still be rendered into the email (if generated) and the failure logged via Pino. Conversely, if the email fails but the RCA was generated, the persisted record MUST still be written where possible (so the RCA is not lost). |

### F. Feature flag & dedup

| # | Requirement |
|---|-------------|
| **FR-19** | The LLM-RCA augmentation MUST be gated behind a **server-side env feature flag** — `CALIBRATION_AUTO_RCA_ENABLED` (server-only; **no** `NEXT_PUBLIC_` prefix), documented in `docs/feature_flags.md` (decided with the user 2026-07-05). When the flag is **off**, the alert path MUST behave exactly as today: deterministic-only email, **no LLM call, no RCA generation, no RCA persistence**. When the flag is **on**, the RCA service runs (still subject to the best-effort/timeout fallback in FR-7–FR-10). The existing `workflow_hash` dedup and the `markAdminAlerted` marker semantics MUST be unchanged whether the flag is on or off. (A DB orchestration flag, if per-tenant rollout is ever needed, is future roadmap — not v1.) |
| **FR-20** | The RCA generation MUST NOT change the existing dedup outcome: the alert is still marked (`markAdminAlerted`) only when the email is actually dispatched, so a transport failure still lets a later run retry. Whether a *later* retry re-generates the RCA (vs. reuses a persisted one) is defined by FR-2 (RCA runs only when the alert is not a dedup-skip) — i.e. a retry that actually sends MAY re-generate. |

### G. Standards & compliance

| # | Requirement |
|---|-------------|
| **FR-21** | All new/changed code MUST comply with the mandatory rules: repository pattern for all DB access (no direct Supabase in the route/service beyond what already exists and is documented), Zod validation at the LLM boundary (FR-15), provider factory for the LLM call (FR-13), structured Pino logging with correlation context (no `console.*`), and TypeScript strict (no implicit `any`; document any unavoidable `any`). |
| **FR-22** | The feature MUST reuse the existing `CalibrationAdminAlertInput` contract and `renderAdminAlertHtml`. The RCA is passed into the alert as an OPTIONAL field on the input (e.g. `autoRca?: CalibrationAutoRca | null`); when absent/null, `renderAdminAlertHtml` renders exactly today's output (guaranteeing the fallback path is byte-compatible with current behaviour and the existing tests still pass). |

### H. Durable correlation key (future log correlation)

| # | Requirement |
|---|-------------|
| **FR-23** | The alert/RCA path MUST persist the request **`correlationId`** (the same `correlationId` already attached to the API-route logs per CLAUDE.md) onto the calibration/execution record at failure time — e.g. as `metadata.correlation_id` on the `calibration_history` row, alongside the `metadata.auto_rca` payload (FR-17), written **through `CalibrationHistoryRepository`** (no direct Supabase write). This is the durable **join key** for any future log correlation (see Out of Scope roadmap item 8 and Open Questions Q6). Persistence MUST be **best-effort and cheap**: it MUST NOT block, delay, or throw out of the calibration tail, and MUST be independent of the feature flag and of whether RCA generation succeeded — the `correlationId` is persisted even when the RCA is not generated (flag off, timeout, or Zod failure), because its purpose is to make *any* failure joinable to logs later. |

### I. Prompt safety — input-value guardrails

| # | Requirement |
|---|-------------|
| **FR-24** | Before any input values are placed into the RCA LLM prompt, the service MUST apply two guardrails to the prompt payload: **(a) truncation** — values exceeding a bounded size (long strings, large arrays/objects) MUST be capped/truncated to a bounded length; and **(b) secret masking** — values matching common secret patterns (API keys, bearer/access tokens, passwords, long high-entropy strings) MUST be masked before being sent to the LLM. This policy applies to the **LLM prompt only**; the existing email embedding of input values (`renderAdminAlertHtml` "Data the agent was processing") is **unchanged** — still internal-only with the "do not forward" footer. The guardrails MUST live in the prompt-builder so they are unit-testable independently of the LLM call (see AC-15). |

## Non-Functional Requirements

| Category | Requirement |
|----------|-------------|
| **Resilience / non-blocking** | RCA generation, persistence, and the augmented send are all best-effort. The calibration tail MUST remain fully isolated — it must never change the route's HTTP response, never throw, and never delay calibration completion or the gate update. Mirrors the existing IMP-2 tail contract. |
| **Performance / time budget** | RCA generation MUST complete within a bounded, **configurable** timeout (code default via the DB-backed config, FR-8) that fits inside the route's `maxDuration = 60s` with headroom for the preceding calibration work and the email send. On timeout, fall back to deterministic-only. One RCA per broken agent version (dedup) bounds latency and cost. |
| **Data sources (v1) — DB-only** | v1 RCA evidence is **durable DB records ONLY** (`calibration_sessions`, `calibration_history`, `agent_executions`, and the `agents` row via `AgentRepository`). Ephemeral platform logs are explicitly **NOT** a v1 data source: (a) a Vercel serverless function cannot read its own or prior invocations' logs — logs are shipped to the platform / log-drains, not a readable file, and the filesystem is ephemeral (`/tmp` only); and (b) Vercel runtime logs are short-retention / ephemeral. The local `dev.log` file is a **testing / repro aid only** (local dev) and MUST NEVER be treated as a production data source. Future log correlation is enabled solely by persisting the `correlationId` (FR-23) as a join key — the log query itself is roadmap, not v1 (see Out of Scope item 8, Open Questions Q6). |
| **Privacy / security** | The RCA LLM call sends **customer data** (the agent's input values + failure details + the workflow definition read per FR-12) to the LLM provider. This MUST be called out explicitly. The alert remains **INTERNAL-ONLY** (the existing "do not forward" footer is retained). The concrete redaction policy for the prompt is **FR-24**: input values are **truncated when oversized** and **masked when they match common secret patterns** before entering the RCA LLM prompt (the email's embedded input-value section is unchanged, still internal-only). Admin recipients MUST be resolved via `AdminAccessService.listAdminEmails()` — never `profiles.role`. The email transport stays system-only (`sendTransactionalEmail` with no `ownerUserId`). |
| **Cost control** | LLM spend MUST be bounded: gated by the feature flag, deduped by `workflow_hash` (one generation per broken version), and single-call (one structured RCA per failure — no multi-turn agent loop). Model/provider selection is DB-config-driven so cost can be tuned without code change. |
| **Provider abstraction** | No hardcoded model names anywhere; all model/provider selection via the provider factory + DB-backed config-with-defaults (FR-13; CLAUDE.md mandatory rule 5). |
| **No plugin-specific hardcoding** | The RCA prompt/logic MUST NOT encode plugin-specific rules or operation names; it reasons from the persisted evidence (including the workflow definition) + plugin schemas as the source of truth (Platform Design Principles). |
| **Backwards compatibility** | With the flag off (or RCA absent), the email and all existing `calibrationAdminAlert.test.ts` expectations MUST remain unchanged. |
| **Logging compliance** | Both files at the integration surface (`lib/calibration/calibrationAdminAlert.ts`, `app/api/v2/calibrate/batch/route.ts`) already use Pino (`createLogger`) — no `console.*` present to convert. The new RCA service MUST use `createLogger` from the outset. (Note: the mirrored config service `AgentGenerationConfigService.ts` uses `console.*` internally; the RCA's own config accessor MUST use Pino, not copy that shortcut.) |

## Acceptance Criteria

QA can verify each of the following:

- [ ] **AC-1** With the feature flag **ON** and a background calibration that lands on `failed`/`needs_review`, the admin alert email contains a distinct **Automated RCA** section that includes all 8 fields (symptom, evidence, earliest failing step + cascade, root-cause layer, root cause, named fix-owner, suggested solutions, remediation path) and a root-cause **layer that is one of the 5 allowed values**.
- [ ] **AC-2** The RCA section is **additive**: the email still contains today's deterministic content — the agent/run table, remaining issues, the `npx tsx scripts/dump-calibration.ts <agentId>` command, the runbook pointer, and the internal-only "Do not forward" footer.
- [ ] **AC-3 (graceful fallback — RCA error)** When RCA generation throws or returns Zod-invalid output, the alert is STILL sent with deterministic-only content (no RCA section), and the failure is logged via Pino. The email is not lost.
- [ ] **AC-4 (graceful fallback — timeout)** When RCA generation exceeds the configured timeout, the code proceeds to send the deterministic alert; total tail time stays within the route budget; calibration completion and the route response are unaffected.
- [ ] **AC-5 (persistence)** After a successful RCA generation, the structured RCA (8 fields + layer) is retrievable from `calibration_history.metadata.auto_rca` via the repository, with a generated-at timestamp and the model/provider used.
- [ ] **AC-6 (provider factory + DB config, no hardcoded model)** The RCA service obtains its LLM via `getProviderFactory()`, and its provider/model/temperature come from the DB-backed config service (`system_settings_config`, cached, with a code `DEFAULT_CONFIG` fallback) — grep shows **no hardcoded model-name string literal** and **no static `getDefaultModel(...)`** in the service; the model actually used is recorded (AC-5).
- [ ] **AC-7 (repository pattern + two-dump evidence)** The RCA service reads its evidence through repositories only (no direct `supabase.from(...)` in the service): the calibration outcome (`calibration_sessions`, `calibration_history`, `agent_executions`) via the calibration repositories, **and** the failed agent's workflow definition — `pilot_steps`, `input_schema`, and the Enhanced Prompt / `user_prompt` — via `AgentRepository.findById(agentId, userId)`. The service does NOT invoke `scripts/dump-calibration.ts` or `scripts/dump-agent.ts`.
- [ ] **AC-8 (Zod validation)** A unit test proves that malformed LLM output fails the Zod schema and the service returns a typed "no RCA" result (which drives the AC-3 fallback).
- [ ] **AC-9 (feature flag off)** With the flag **OFF**, no LLM call is made, no RCA section appears, no `auto_rca` record is written, and the email is byte-compatible with today's output (existing `calibrationAdminAlert.test.ts` still passes unchanged).
- [ ] **AC-10 (dedup preserved)** A second background failure for the same `(agentId, userId, workflowHash)` that hits the dedup path (`hasAdminAlertBeenSent === true`) does NOT generate a new RCA and does NOT send a second email — one RCA + one alert per broken agent version.
- [ ] **AC-11 (never throws)** Forcing an exception inside RCA generation, persistence, and the send does not throw out of the calibration `finally` tail, does not change the HTTP response, and does not leave the agent stuck on "running" (gate is still set).
- [ ] **AC-12 (escaping/privacy)** Malicious content in issue text / input values / workflow-definition / LLM output is HTML-escaped in the RCA section (no `<script>` survives), and the email retains the internal-only footer. Admin recipients came from `AdminAccessService.listAdminEmails()`.
- [ ] **AC-13 (correlationId persisted)** After a background failure, the request `correlationId` is persisted on the calibration record (e.g. `calibration_history.metadata.correlation_id`) via the repository — and is present **even when the flag is off or the RCA was not generated** (timeout / Zod failure), because it is the durable log-join key (FR-23).
- [ ] **AC-14 (test coverage)** Integration/unit tests cover: happy path (RCA rendered, incl. workflow-definition read), RCA-failure fallback, timeout fallback, flag-off deterministic-only, dedup skip, and correlationId persistence — at minimum the happy path and one failure path per the testing standard.
- [ ] **AC-15 (prompt redaction)** A unit test over the prompt-builder proves that input values matching common secret patterns are **masked** and oversized values are **truncated** in the RCA LLM prompt, while the email's embedded input-value section ("Data the agent was processing") is unaffected (FR-24).

## Out of Scope / Future Roadmap

| # | Item | Note |
|---|------|------|
| 1 | RCA for runtime executions, agent creation, or V6 generation failures | v1 hooks calibration failures only (the existing admin-alert path). Other lifecycle stages are future roadmap. |
| 2 | Multi-turn / agentic RCA (tool-using loop like the TS Claude subagent) | v1 is a single structured LLM call over persisted evidence. The TS Claude-Code subagent cannot run on Vercel; this productionizes the *method*, not the agent. |
| 3 | Auto-triggering a fix (BA/SA/Dev) from the RCA | Out of scope — the RCA is delivered to admins/TL; TL routes, exactly as for a TS conclusion. |
| 4 | Following the 3-skill handoff chain (calibration → creation → V6) at runtime | v1 produces a single calibration-scoped RCA (classified into one of the 5 layers). Cross-stage chaining stays with the human TS/TL. |
| 5 | A UI to view/link persisted RCAs (e.g. from the Admin Agent Health Dashboard) | Persistence (FR-17) enables this; the UI itself is roadmap. A dedicated RCA column/table (vs. the `metadata.auto_rca` key) is revisited here if the dashboard needs to query RCAs at scale. See `ADMIN_AGENT_HEALTH_DASHBOARD_REQUIREMENT.md`. |
| 6 | Re-running / reproducing the agent to diagnose | Prohibited — RCA uses persisted evidence only (non-determinism trap), same as the `calibration-rca` method. |
| 7 | Localizing / user-facing translation of the RCA | The RCA is internal/admin-facing technical content; no user-facing translation. |
| 8 | **Log-drain / queryable-log-store integration** (e.g. Axiom, Datadog, Better Stack) that lets the RCA service query platform logs by `correlationId` during generation | **Confirmed-deferred to v2** (user: "future requirement"), and **gated on a log drain actually existing** (Vercel runtime logs are ephemeral and unqueryable by the function itself — see the Data-sources NFR). The `correlationId` persistence (FR-23) is what makes this possible later — it is the join key. Trade-offs to weigh before building: added **cost** (log-store subscription + query volume), added **latency** inside the 60s `maxDuration` budget (a network round-trip to the log store per RCA), and a **new failure mode** (the RCA now depends on an external log API — must stay best-effort per FR-7). See Open Questions Q6. |

## Open Questions

All questions are resolved or confirmed-deferred — none remain blocking. Ready for TL/user sign-off → SA workplan review.

- [x] **Q1 — Model + timeout — RESOLVED (user, 2026-07-05)**. The RCA resolves its **provider/model/temperature** *and* its **generation timeout** from a **DB-backed config-with-defaults mechanism** — a dedicated calibration-RCA entry in the `system_settings_config` table, read through a small cached accessor (~5-min in-memory TTL) with a hardcoded code-level `DEFAULT_CONFIG` fallback — mirroring `lib/agentkit/v6/config/AgentGenerationConfigService.ts`. It does **NOT** use a static `factory.getDefaultModel(...)` or an env value for model selection. Code-default fallback: a **mid-tier reasoning model**; the timeout is a configurable value with a sensible code default, bounded to fit inside `maxDuration = 60s` with headroom. The LLM call still routes through the provider factory. Encoded in **FR-13** (model/provider/temperature) and **FR-8** (timeout); verified by **AC-6**.
- [x] **Q2 — Persistence target — RESOLVED (user, 2026-07-05)**. Persist the RCA as `calibration_history.metadata.auto_rca` (JSON) via `CalibrationHistoryRepository` — no migration, reuses the existing row + repo. A dedicated column/table is revisited **only if** a future dashboard needs to query RCAs at scale (roadmap — Out of Scope item 5). Encoded in **FR-17**; the `correlationId` (FR-23) rides the same `metadata` write.
- [x] **Q3 — Input-value handling — RESOLVED (user, 2026-07-05)**. Include the input values in the RCA prompt (they are already in the internal-only email) **with guardrails**: **truncate** oversized values and **mask** values matching common secret patterns before they enter the LLM prompt. Applies to the **LLM prompt only** — the email's embedded input-value section is unchanged (still internal-only, "do not forward"). Encoded in **FR-24** and the Privacy/Security NFR; verified by **AC-15**.
- [x] **Q4 — Feature-flag mechanism — RESOLVED (user, 2026-07-05)**. Use a **server-side env flag** `CALIBRATION_AUTO_RCA_ENABLED` for v1 (server-only, no `NEXT_PUBLIC_`), documented in `docs/feature_flags.md`. A DB orchestration flag is deferred to future roadmap if per-tenant rollout is later required. Encoded in **FR-19**.
- [x] **Q5 — Layer set — RESOLVED (user, 2026-07-05)**. Keep the **5-value TS layer set** (input/data · V6 generation · runtime/external API · calibration-detection · creation chat flow) so the RCA is routable identically to a Troubleshooter conclusion — a calibration-only failure will almost always classify into the 4 calibration-scoped values, with *creation chat flow* available when the earliest cause traces to creation. Already encoded in **FR-15**.
- [x] **Q6 — Log-drain integration — CONFIRMED-DEFERRED (user, 2026-07-05: "future requirement")**. Explicitly **not v1** — a future/roadmap item (Out of Scope item 8). v1 persists the `correlationId` (FR-23) as the durable log-join key only; the actual log query is roadmap, gated on a log drain existing and justified against the cost / latency / 60s-budget / new-failure-mode trade-offs, and kept best-effort (FR-7) if adopted. **Relationship to the Troubleshooter (TS) requirement (`docs/requirements/TROUBLESHOOTER_AGENT_REQUIREMENT.md`):**
This feature **shares the methodology and the conclusion shape** with TS but runs on a **different execution surface**. Both apply the `calibration-rca` 6-step method — over the same two evidence sources (calibration outcome + agent workflow definition) — and produce the same **8-field conclusion + one of the 5 root-cause layers**. The difference: TS is a **Claude-Code subagent** that runs in the dev/agent environment (Read/Bash/Write, the `dump-calibration.ts` + `dump-agent.ts` scripts, writes a doc under `docs/investigations/`) and is strictly diagnostic-and-recommend; this feature is a **production, in-Vercel, LLM-driven service** that runs at calibration-failure time, reads the same evidence via repositories (the runtime equivalent of the two dumps), and delivers the RCA in the admin email + a persisted record. TS cannot run inside the serverless app — hence this is a separate in-product implementation that **reuses the method**, not a call into the TS agent. The RCA it emits is intentionally **routable by the TL exactly like a TS conclusion**.
