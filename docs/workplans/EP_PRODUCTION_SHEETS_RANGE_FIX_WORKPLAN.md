> **Scope note (2026-07-05):** originally the Sheets-range fix; broadened to the **general EP required-plugin-param class** after the `df67bf69` topic-drop proved the same root (a required plugin input supplied by the user but not reliably materialized into the EP). Sheet1 and topic are the two proven instances. Parts 1–2 = Sheet1; **Part 3 = the generalization + the deterministic V6 backstop**.

## Overview

Root cause (proven from thread `08c05035…` `iterations[]`): v16 **Phase 3** fabricated `range="Sheet1"` from a `gid=0` spreadsheet URL — the narrative preserved "gid=0 / first tab" while the structured `resolved_user_inputs` collapsed it to Google's default tab name. Drivers: `range` mis-framed as "a single sheet tab name" (v16 L275), a `Sheet1!`-leading plugin example, and a mandate to emit a concrete value, with **no rule** for opaque-id / URL-fragment references.

**Deeper root cause — a v16 regression (verified 2026-07-02).** The mis-framing was **latent in v14/v15** and harmless there, because those versions' *"Ask until nothing is ambiguous"* bias + the parameter-aware block **elicited the tab from the user** (v15 GOOD example: *"Which single sheet tab should rows be appended to?"*). v16 added the **PACING & CONVERGENCE / STOP-early** rule (commit `5ac952c`, OI1) — verified new to v16 (`git`: "PACING & CONVERGENCE" 0/0/1 across v14/v15/v16). That flipped the bias from *ask* to *default-and-stop*, so the tab question got **skipped**, leaving Phase 3 to materialize a value it never collected → fabrication. **The pacing/assumptions requirement is the trigger; the mis-framing is the latent fault it activated.**

**Three fixes, layered (primary-first):**
- **#3 (PRIMARY — pacing carve-out, Phase 2):** restore the v15 behavior *surgically* — required plugin parameters are not silently defaultable; ask/confirm a spec-matching value. Applied + smoke-tested (below).
- **#1 (BACKSTOP — Phase 3 anti-fabrication):** if the value still isn't collected, don't fabricate — emit name-less/defer.
- **#2 (HYGIENE — framing):** remove the false "single sheet tab name" teaching.
- Deterministic guarantee = the **optional** gid-carry cycle (see Part 2), now demoted.

### FR12 finding → prompt fixes are a strong mitigation, not a guarantee
Prompt tuning is **non-deterministic** here — both approaches top out around 4/6: the Phase-2 carve-out (#3) reliably asks the tab **when the plugin param is the fresh topic** (4/6 right after the URL) but not on a late catch-up (0/6); the Phase-3 backstop (#1) emits name-less/defer 4/6. Combined ≈ ~90% clean vs a 100%-`"Sheet1"` baseline. **An LLM slot cannot be made deterministically correct by prompt text alone.** The deterministic guarantee therefore lives at a layer that can read the live sheet — see Part 2.

## Part 4 — Backward compatibility: runtime resolver for EXISTING broken agents (topic-class "Option A") — ⏸ DEFERRED (population measured 2026-07-06 — too small to warrant the build)

> **DECISION (2026-07-07): DO NOT BUILD the runtime resolver — DEFERRED unless the population grows.** The SA design review (below) returned 🔄 Revision Required with three plumbing blockers (C1–C3) *and* gated the whole thing (Q1/Q4) on **measuring the closed broken population first**. That measurement is now done and settles it:
>
> | Measure (82 non-deleted agents scanned, 2026-07-06) | Count |
> |---|---|
> | **Exact topic-drop class** (`chatgpt-research/research_topic` missing `topic`) | **2 — both `draft`** (`c6e8a3d6` 2025-10-29, `df67bf69` 2026-07-05) |
> | Broad upper bound (any action step with `params:{}`) | ≤ 8 (mixed draft/active; upper bound, many likely benign) |

> The exact class is **2 agents, both drafts** (one is the test agent `df67bf69`) — essentially zero production exposure. Per the SA Q4 ruling (*small/closed → migration-primary, resolver as safety-net*), standing up the C1/C2 engine changes (a new generic-resolver tier + detector wiring + the `add_missing_parameter` boundary) is far more work than the problem warrants. **The forward safety net is already P3.1** (catches new agents at compile); the resolver's only non-redundant value was the existing population, which is 2 drafts. **Resolution:** handle the existing set by **regenerate / recreate** (they're drafts); rely on P3.1/P3.2 for all new agents. Revisit Part 4 only if this class starts appearing in *active/production* agents at volume. The design + SA review below are retained as the ready-to-resume spec if that happens.

**Why Part 4 exists (the gap Part 3 leaves).** Every Part 3 layer is **upstream** — the v16 prompt (P3.3), the IR converter (P3.2), the compiler (P3.1) — so they run **only at generation** and repair only **newly generated / regenerated** agents. An agent already persisted with `step.params = {}` executes its **stored** DSL on every run/calibration (V6 never re-runs), so P3.1/P3.2 never touch it — it fails forever with `Topic is required for research` (and, pre-`2f40172`, emailed an empty digest each calibration). The Sheet1 class already has a runtime backward-compat layer — **Part 2 Option A** (`googleSheetsRange.ts`, fires at calibration on the stored DSL). The **topic-drop class has none.** Part 4 is that missing layer: the topic-class analog of Option A.

**The scope decision this needs from SA (gating everything below):** *do we repair agents already in the DB, or only prevent new ones?* Part 3 alone suffices for new agents; build Part 4 only if in-place repair of the existing broken population is in scope.

**Design — a new parameter resolver in the existing family.**
- **File:** `lib/pilot/shadow/parameterResolvers/descriptionDerivedParam.ts` (sibling of `googleSheetsRange.ts`), registered in `ParameterResolverEngine`, route-wired exactly like Option A.
- **Trigger:** the calibration `ConstrainedSemanticValidator` already flags `Missing required parameter: <p>` statically (`dev.log` L5203) / runtime `parameter_error`. Part 4 **acts on** that finding — it does not add a second detector (P3-R4 discipline: wire the existing signal).
- **Eligibility gate (the make-or-break — the fabrication boundary).** Fire **only** for **descriptive free-text params** whose value the step description would naturally contain (topic, query, subject, instruction, prompt). Decided **schema-driven** — `type:string` + `required` + descriptive semantic-type — and **exactly one** unbound such param (mirror P3.2/B2's single-target rule; ≥2 → defer, don't guess). **Never** fire for opaque identifiers / locators (ids, tab names, ranges, emails, folder ids): those are the Sheet1 / gid-carry lane (live-resolve or ask). Extracting an identifier from a description would **fabricate** — Anti-pattern C, the Sheet1 sibling. **Zero plugin-name branches** (grep-gated, per B2).
- **Recovery source — extraction, NOT fabrication (the key distinction from Sheet1).** Read the persisted step `description`/`name` (which still carries the subject verbatim, e.g. *"Research the web… about retail solutions in Israel"*) and derive the value. This is **recovery of an already-present value** — categorically different from the Sheet1 guess, where the correct tab name existed *nowhere the runtime could see*. Deterministic where possible; a **bounded, confidence-scored LLM extraction** ("what is this step's {param}?") is the semantic step only when deterministic parsing can't.

- **Deterministic-first** (Part 3's thesis): prefer deterministic recovery from the description before any LLM call; the LLM is the narrow, threshold-gated semantic extractor, not the default.

### Architectural assessment

**Deterministic-first is correct — and the empirical replay makes it non-negotiable.** §5A of the RCA proves prompt-only structuring is non-deterministic in *both* prompt versions (v15 4/5, v16 3/5). **P3.1 and P3.2 are complementary, not redundant — keep both.** They address *different* points on the failure curve:
- **P3.2 (bind)** is the *resolve* — it deterministically populates `topic` from the search subject the intent already carries, so the common case never reaches a guard. Critically, the compiler already has the scaffolding: `detectNullableToRequiredMappings` (O16, `ExecutionGraphCompiler.ts` L3902-3999) already iterates every action step, fetches `getActionInputSchema(step.plugin, step.operation)`, builds `requiredParams = new Set(pluginSchema.required || [])`, reads `params = (step as any).params || step.config || {}`, and recurses into scatter/conditional. **P3.1 is a sibling pass at that exact site** — for each `requiredParam` absent-or-empty in `params`, escalate. ### Rulings on the 5 open questions

**Q1 — P3.1 failure mode → emit a structured `user_inputs_required` marker as the DEFAULT; hard-fail ONLY when the value was bindable-but-lost. APPROVED (the workplan's recommendation, with a sharpened boundary).** The decision hinges on *why* the param is empty:
- **Genuinely absent** (the user never supplied it): surface a `user_inputs_required` item the creation/review flow presents ("What topic should this research?") — degrade gracefully, never hard-block agent creation. This aligns with Principle 11 (make the failure *visible*, don't hide it) *and* with the calibration RCA's honest-failure posture — a caught generation-time gap that asks the user beats a silent `params:{}` that emails an empty digest.
- **Present-but-lost** (the value exists in the intent/EP but the converter failed to bind it — exactly the `df67bf69` case, where the subject *was* in `query`): this is a **compiler/converter bug**, and after P3.2 lands it should never happen. If P3.1 still fires on a value that P3.2 should have bound, that is a genuine defect and hard-fail-the-compile is correct (fail loud, surface in generation logs, do not ship a broken agent). Rationale: the two sub-cases have different owners — absent = user-input gap (ask); lost = pipeline bug (fail loud so we catch it). A blanket hard-fail would block legitimate agent creation whenever the user under-specified; a blanket marker would mask real converter regressions as "ask the user." **Distinguish on provenance: did the intent/EP carry a candidate value for this param? If yes and it's still empty → hard-fail (bug). If no → `user_inputs_required` (ask).** This also satisfies Principle 2 (fail loud, never substitute a default — note: **do not** let P3.1 auto-fill a placeholder; that would be Anti-pattern C, the exact Sheet1 fabrication sibling).

**Q2 — P3.2 scope → gate strictly on schema `required` + string type; NO "primary subject" heuristic. APPROVED.** Keep it schema-driven and plugin-agnostic. The binding rule: *when the bound action's schema declares a required param of `type: string` that is unbound after normal mapping, and the intent carries a free-text search subject (today `query`/summary), bind the subject to that required string param.* Constrain it further to avoid false binds:
- Only when the action has **exactly one** unbound required string param (unambiguous target). If two required string params are unbound, that is ambiguity → defer to P3.1's `user_inputs_required` marker, do not guess which slot the subject fills (guessing here would be the semantic-ambiguity anti-pattern SA rejects).
- Respect `minLength`/`maxLength` (`topic` is 3-500) as a sanity gate, not a hard bind condition.
- **Hard guardrail (blocking):** P3.2 must be implemented with **zero** `plugin_key === 'chatgpt-research'` / `action === 'research_topic'` branches. The converter already carries `effectivePluginKey !== 'chatgpt-research'` special-cases at L536/566 — Part 3 must **not add a third**. This is Principle 6 / Anti-pattern F ("Plugin-specific branch in a generic component"). A "primary subject" heuristic is rejected precisely because it invites a plugin-keyed shortcut; the schema `required`+`string`+single-unbound-target rule is fully generic and testable against sheets/gmail/chatgpt-research fixtures without naming any of them. **Add a grep gate (mirroring the gid-workplan A5): a grep for `chatgpt-research` / `research_topic` in the P3.2 diff must return zero new hits.**

### Architectural assessment

**The backward-compat gap is real and correctly diagnosed (Q1 basis).** Verified:

**Three plumbing claims are inaccurate against the tree — these are the real work, and they gate the design.**

- **C1 (blocking) — the trigger signal does NOT currently reach the ParameterResolverEngine.** Part 4 says it "acts on" an existing `parameter_error` / `ConstrainedSemanticValidator` finding and adds "no second detector (P3-R4 discipline)." But: (a) the engine's `planIssue` fires **only** on `issue.category === 'parameter_error'` AND requires `issue.suggestedFix.action.parameterName` + `affectedSteps[0].stepId` (`ParameterResolverEngine.ts` L141-146). (b) The runtime error `"Topic is required for research"` is **not** parsed as a `parameter_error` — `IssueCollector.parseParameterError` (L1129-1246) matches none of its three patterns and the string contains **none** of the config keywords (`not found`/`invalid`/`does not exist`/…), so it returns `null` and no `parameterName` is ever produced. (c) The static "Missing required parameter: <p>" string is **not** emitted by `ConstrainedSemanticValidator` (it has no such string) — it comes from `StructuralRepairEngine` (type `missing_required_parameter`, L470-484) and from V6-generation-time `PluginParameterValidator`. Neither is a `parameter_error`-category, `parameterName`-bearing issue the engine consumes. **Net: as designed, Part 4 would fire on nothing.** The design must specify how a missing-required-param signal is routed into the engine's `parameter_error` channel with `parameterName` populated — and doing so **is** adding/extending a detector, which contradicts the P3-R4 "wire the existing signal" claim. Resolve honestly: either extend `IssueCollector` to categorize a missing-required-param as `parameter_error` (a real, owned change), or integrate with `StructuralRepairEngine` (see C3), and drop the "no second detector" framing.

- **C2 (blocking) — "registered … route-wired exactly like Option A" is not achievable with the current registry.** `ParameterResolverRegistry` keys every resolver by the exact `${plugin}.${action}.${parameter}` triple (`index.ts` L16-30) and `lookup()` is an exact `Map.get`. Part 4 needs a genuine engine/registry extension: a **fallback/generic-resolver tier** evaluated when no exact-key resolver matches (a separate list the engine consults after the keyed lookup misses), with its own `appliesTo` gate carrying the eligibility logic. This is a net-new generic capability in the engine, not a drop-in sibling. Design it explicitly and keep the engine plugin-agnostic (the eligibility/extraction specifics live in the resolver, matching the existing contract).

- **C3 (blocking) — composition analysis omits `StructuralRepairEngine.add_missing_parameter`.** There is already a calibration repair for missing required params: `add_missing_parameter` — "Add missing required parameter **with smart default**" (`StructuralRepairEngine.ts` L83, L422-486). Part 4's P3-R6-extension composition table addresses Option A + the empty-send guard but **not** this. ### Eligibility-gate adversarial analysis (the crux — Q2)

**The extraction-vs-fabrication line is sound in principle but the schema-driven separator Part 4 relies on likely does not exist today.** Adversarial walk:

- **`type:string` + `required` does NOT separate descriptive free-text from opaque identifiers.** A tab name, a folder id, an email, an A1 range, and a research topic are **all** `type:string, required`. So the two proposed positive signals are shared by exactly the class Part 4 must exclude. The gate cannot stand on `type:string`+`required` alone.
- **Is there a schema-driven "descriptive semantic-type" signal?** Part 4 asserts one ("descriptive semantic-type") but names no schema field. I found no `x-semantic-type`/`format` convention carrying free-text-vs-identifier in the plugin definitions inspected. If none exists, the gate collapses to either (a) a **param-name allowlist** (`topic`/`query`/`subject`/`instruction`/`prompt`) — a brittle heuristic that silently mis-handles any plugin whose free-text param is named differently, and which is itself a soft cousin of the plugin-name-branch anti-pattern (Principle 6 / Anti-pattern F) — or (b) an LLM classifier. **A bare name-allowlist as the primary gate is REJECTED** (see Q2 ruling). The defensible path is to gate on **exclusion markers that already exist** in JSON schema — `pattern` present, `enum` present, `format: email|uri|id`, a tight `maxLength` — treat any of those as "identifier-like → exclude," and only *positively* include a param that (i) is `required` `string`, (ii) carries **none** of those identifier markers, (iii) has a wide free-text length range (e.g. `minLength` small, `maxLength` large like topic's 3–500), and (iv) is the **single** unbound required string param (mirror P3.2/B2). If that positive test can't be met from the schema, the resolver must **defer to `needs_review`, never guess** — conservative-by-default.
- **Is "extraction from the description" different-in-kind from the Sheet1 guess? Yes — but only under a verifiability guard.** Sheet1's correct tab name existed *nowhere the runtime could see* (the `gid` was discarded), so Option A had to read the live sheet and *choose among tabs* = a genuine guess. The topic is present **verbatim** in `step.description`. Recovering a present token ≠ fabricating an absent one — the distinction holds. **But** a bounded LLM "what is this step's topic?" can paraphrase, mis-scope, or invent. So extraction is non-fabrication **only if** the recovered value is **verifiably grounded** in the description text (a substring / derivable span, not a free invention). Make grounding an explicit acceptance criterion: an extracted value that is not traceable to the description is rejected regardless of confidence. With that guard the boundary is defensible; without it, "extraction" quietly re-imports fabrication under a new name.

Decision rule: measure → small/closed = migration-primary + resolver-safety-net; large/open = resolver-primary.
