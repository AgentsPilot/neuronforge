> **Scope note (2026-07-05):** originally the Sheets-range fix; broadened to the **general EP required-plugin-param class** after the `df67bf69` topic-drop proved the same root (a required plugin input supplied by the user but not reliably materialized into the EP). Sheet1 and topic are the two proven instances. Parts 1–2 = Sheet1; **Part 3 = the generalization + the deterministic V6 backstop**.

## Overview

Root cause (proven from thread `08c05035…` `iterations[]`): v16 **Phase 3** fabricated `range="Sheet1"` from a `gid=0` spreadsheet URL — the narrative preserved "gid=0 / first tab" while the structured `resolved_user_inputs` collapsed it to Google's default tab name. Drivers: `range` mis-framed as "a single sheet tab name" (v16 L275), a `Sheet1!`-leading plugin example, and a mandate to emit a concrete value, with **no rule** for opaque-id / URL-fragment references.

**Deeper root cause — a v16 regression (verified 2026-07-02).** The mis-framing was **latent in v14/v15** and harmless there, because those versions' *"Ask until nothing is ambiguous"* bias + the parameter-aware block **elicited the tab from the user** (v15 GOOD example: *"Which single sheet tab should rows be appended to?"*). v16 added the **PACING & CONVERGENCE / STOP-early** rule (commit `5ac952c`, OI1) — verified new to v16 (`git`: "PACING & CONVERGENCE" 0/0/1 across v14/v15/v16). That flipped the bias from *ask* to *default-and-stop*, so the tab question got **skipped**, leaving Phase 3 to materialize a value it never collected → fabrication. **The pacing/assumptions requirement is the trigger; the mis-framing is the latent fault it activated.**

**Three fixes, layered (primary-first):**
- **#3 (PRIMARY — pacing carve-out, Phase 2):** restore the v15 behavior *surgically* — required plugin parameters are not silently defaultable; ask/confirm a spec-matching value. Applied + smoke-tested (below).
- **#1 (BACKSTOP — Phase 3 anti-fabrication):** if the value still isn't collected, don't fabricate — emit name-less/defer.
- **#2 (HYGIENE — framing):** remove the false "single sheet tab name" teaching.
- Deterministic guarantee = the **optional** gid-carry cycle (see Part 2), now demoted.

### FR12 finding → prompt fixes are a strong mitigation, not a guarantee
Prompt tuning is **non-deterministic** here — both approaches top out around 4/6: the Phase-2 carve-out (#3) reliably asks the tab **when the plugin param is the fresh topic** (4/6 right after the URL) but not on a late catch-up (0/6); the Phase-3 backstop (#1) emits name-less/defer 4/6. Combined ≈ ~90% clean vs a 100%-`"Sheet1"` baseline. **An LLM slot cannot be made deterministically correct by prompt text alone.** The deterministic guarantee therefore lives at a layer that can read the live sheet — see Part 2.

## Part 3 — Generalize to all required plugin params + deterministic V6 backstop (topic-drop) — ⬜ SCOPED, awaiting SA

**Origin:** [V6_RCA_df67bf69_missing-topic.md](../investigations/V6_RCA_df67bf69_missing-topic.md). Agent `df67bf69`'s research subject ("retail solutions in Israel") was supplied verbatim in the prompt but **not structured** into the EP → compiled `step1.params = {}` → runtime `Topic is required for research` → empty digest emailed.

**Same class as Sheet1, different mode:** Sheet1 *fabricated* a value for a slot it recognized; topic was *dropped* because a search **subject** matches none of the EP's surfacing patterns (identifiers / recipients). **Empirically proven non-deterministic in BOTH prompt versions** — a df67bf69-conversation replay structured the topic **v15: 4/5, v16: 3/5** (§5A of the RCA). So this is **not** fixable by prompt tuning or reverting to v15; it needs a **deterministic downstream layer** (the identical lesson to Sheet1: prompt tops out ~60–80%, the guarantee lives downstream).

### Fix layers (deterministic-first — re-prioritized from the RCA §7 per the replay data)

| # | Layer | Owner | Change | Priority |
|---|---|---|---|---|
| **P3.1** | **No-empty-required-params compile guard** | v6-pipeline (`ExecutionGraphCompiler`) | Never emit an `action`/`data_source` step whose `params` omit a field the plugin declares `required`. If unbound after binding → **fail the compile with a specific error**, not `params:{}`. Plugin-agnostic; the generic-compiler-optimization class CLAUDE.md permits. Converts a silent runtime failure + empty email into a caught generation error. | **PRIMARY (deterministic safety net)** |
| **P3.2** | **Subject → required-text-param binding** | v6-pipeline (`IntentToIRConverter`) | Today a search subject → `params.query` ([L376](../../lib/agentkit/v6/compiler/IntentToIRConverter.ts#L376)). The subject is always present (intent `query`/summary), so this resolves the common case deterministically. | **PRIMARY (deterministic resolve)** |
| **P3.3** | **EP required-param surfacing rule** | agent-creation (v16 Phase 3) | Generalize the existing "surface resource identifiers into `resolved_user_inputs`" rule (L496, L502) to **"surface ALL required params of bound actions, including a search/analysis subject"** — as a non-negotiable that the v16 "infer/defaults" bias may not skip. ### Architectural assessment

**Deterministic-first is correct — and the empirical replay makes it non-negotiable.** §5A of the RCA proves prompt-only structuring is non-deterministic in *both* prompt versions (v15 4/5, v16 3/5). That is the same shape as the Sheet1 FR12 finding (prompt tops out ~4/6). A required plugin input that the user genuinely supplied must not depend on an LLM coin-flip to reach `params`. So P3.3 (the prompt surfacing rule) is correctly demoted to "quality, rides on top" and P3.1/P3.2 correctly carry the guarantee. Reverting to v15 is correctly rejected (v15 is only 80%, not 100%).

**P3.1 and P3.2 are complementary, not redundant — keep both.** They address *different* points on the failure curve:
- **P3.2 (bind)** is the *resolve* — it deterministically populates `topic` from the search subject the intent already carries, so the common case never reaches a guard. Without it, P3.1 alone would only ever *fail loudly*, never *fix*, on the single most common research-agent shape — a strict regression in UX (every research agent would hard-fail or need user input).
- **P3.1 (guard)** is the *safety net* — it catches every future/unforeseen drop (any plugin, any required param) that P3.2's specific binding doesn't cover, and converts a silent runtime failure + empty email into a caught generation signal. It is the plugin-agnostic backstop; P3.2 is the targeted resolve.

Neither is sufficient alone. P3.2-without-P3.1 leaves the class open for every other required param (Gmail `to`, etc.); P3.1-without-P3.2 turns the *known-good* research case into a hard-fail/needs-input. **Ship both. Order P3.2 before P3.1 (Q3).**

**P3.1 belongs in the permitted "generic compiler optimization" class — confirmed, with a concrete home.** The no-empty-required-params guard is exactly the class CLAUDE.md permits (scales to any plugin, reads the schema as source of truth). Critically, the compiler already has the scaffolding: `detectNullableToRequiredMappings` (O16, `ExecutionGraphCompiler.ts` L3902-3999) already iterates every action step, fetches `getActionInputSchema(step.plugin, step.operation)`, builds `requiredParams = new Set(pluginSchema.required || [])`, reads `params = (step as any).params || step.config || {}`, and recurses into scatter/conditional. **P3.1 is a sibling pass at that exact site** — for each `requiredParam` absent-or-empty in `params`, escalate. This is the strongest evidence P3.1 stays generic: it reuses the existing schema-driven, plugin-agnostic O16 machinery and adds zero plugin-keyed branches.

### Rulings on the 5 open questions

**Q1 — P3.1 failure mode → emit a structured `user_inputs_required` marker as the DEFAULT; hard-fail ONLY when the value was bindable-but-lost. APPROVED (the workplan's recommendation, with a sharpened boundary).** The decision hinges on *why* the param is empty:
- **Genuinely absent** (the user never supplied it): surface a `user_inputs_required` item the creation/review flow presents ("What topic should this research?") — degrade gracefully, never hard-block agent creation. This aligns with Principle 11 (make the failure *visible*, don't hide it) *and* with the calibration RCA's honest-failure posture — a caught generation-time gap that asks the user beats a silent `params:{}` that emails an empty digest.
- **Present-but-lost** (the value exists in the intent/EP but the converter failed to bind it — exactly the `df67bf69` case, where the subject *was* in `query`): this is a **compiler/converter bug**, and after P3.2 lands it should never happen. If P3.1 still fires on a value that P3.2 should have bound, that is a genuine defect and hard-fail-the-compile is correct (fail loud, surface in generation logs, do not ship a broken agent). Rationale: the two sub-cases have different owners — absent = user-input gap (ask); lost = pipeline bug (fail loud so we catch it). A blanket hard-fail would block legitimate agent creation whenever the user under-specified; a blanket marker would mask real converter regressions as "ask the user." **Distinguish on provenance: did the intent/EP carry a candidate value for this param? If yes and it's still empty → hard-fail (bug). If no → `user_inputs_required` (ask).** This also satisfies Principle 2 (fail loud, never substitute a default — note: **do not** let P3.1 auto-fill a placeholder; that would be Anti-pattern C, the exact Sheet1 fabrication sibling).

**Q2 — P3.2 scope → gate strictly on schema `required` + string type; NO "primary subject" heuristic. APPROVED.** Keep it schema-driven and plugin-agnostic. The binding rule: *when the bound action's schema declares a required param of `type: string` that is unbound after normal mapping, and the intent carries a free-text search subject (today `query`/summary), bind the subject to that required string param.* Constrain it further to avoid false binds:
- Only when the action has **exactly one** unbound required string param (unambiguous target). If two required string params are unbound, that is ambiguity → defer to P3.1's `user_inputs_required` marker, do not guess which slot the subject fills (guessing here would be the semantic-ambiguity anti-pattern SA rejects).
- Respect `minLength`/`maxLength` (`topic` is 3-500) as a sanity gate, not a hard bind condition.
- **Hard guardrail (blocking):** P3.2 must be implemented with **zero** `plugin_key === 'chatgpt-research'` / `action === 'research_topic'` branches. The converter already carries `effectivePluginKey !== 'chatgpt-research'` special-cases at L536/566 — Part 3 must **not add a third**. This is Principle 6 / Anti-pattern F ("Plugin-specific branch in a generic component"). A "primary subject" heuristic is rejected precisely because it invites a plugin-keyed shortcut; the schema `required`+`string`+single-unbound-target rule is fully generic and testable against sheets/gmail/chatgpt-research fixtures without naming any of them. **Add a grep gate (mirroring the gid-workplan A5): a grep for `chatgpt-research` / `research_topic` in the P3.2 diff must return zero new hits.**

**Q3 — Ordering → P3.2 (bind) runs before P3.1 (guard). APPROVED.** P3.2 is a converter-phase resolve; P3.1 is a compiler-phase backstop — they are already in the correct pipeline order (IR conversion precedes compilation), so this is naturally satisfied by *where* each lives. The guard then fires only on genuinely-unbindable values, which is exactly what Q1's marker-vs-fail distinction keys off. Make the invariant explicit in a code comment at the P3.1 site: "by this point P3.2 has bound every subject→required-string param it can; a remaining empty required param is either a user-input gap (→ marker) or a converter bug (→ fail)."

**Q4 — P3.4 grouping → SPLIT it into its own tracked item; do NOT block Part 3's primary fix on it. APPROVED with a caveat.** P3.4 is a *different* defect (EP over-promising output fields the action doesn't return → transform-shape + phantom-field bugs), with a *different* owner (EP data-contract fidelity + transform grounding) and a *different* fix mechanism (output-schema grounding, not required-param binding). Folding it in risks scope-creeping the deterministic fix that is the priority. **However** — the two are causally linked (same mis-modeled `research_topic` capability) and the regression scenario for Part 3 will re-generate `df67bf69`, which *also* exercises step2/3/4. So: keep P3.4 as a **named sub-item in this cycle's tracklist but sequenced after P3.1/P3.2**, and if it grows beyond a one-EP-rule + one-grounding-check it graduates to its own WEAK_POINTS WP. Do not let steps 2-5 cascade defects gate the P3.1/P3.2 landing — once `topic` binds, the cascade stops and P3.4 becomes independently testable.

**Q5 — Regression-scenario home → `tests/v6-regression/scenarios/<name>/` (NOT `scripts/test-dsl-execution-simulator/`). CORRECTED.** The workplan (and the terse V6-Work-Protocol pointer) says "add under `scripts/test-dsl-execution-simulator/`" — that directory holds the simulator **engine** (`dsl-simulator.ts`, `stub-data-generator.ts`, `__tests__/`), not scenario snapshots. Per Principle 12, the canonical snapshot home is **`tests/v6-regression/scenarios/<name>/`**. I verified the required file set from an existing scenario (`complaint-email-logger/`): `scenario.json`, `enhanced-prompt.json`, `intent-contract.json`, `phase2-data-schema.json`, `phase4-pilot-dsl-steps.json`, `phase4-workflow-config.json`, plus `output*` dirs. **Add a `research-topic-required-param` (name Dev's call) scenario under `tests/v6-regression/scenarios/` with that exact file set**, snapshot the post-fix `phase4-pilot-dsl-steps.json` showing `step1.params.topic` bound, and set `scenario.json`'s `phase_e_success`/`phase_e_caveat`. Because Phase-1/Phase-3 are non-deterministic (§5A), the assertion must be replay-based against the committed EP/intent-contract snapshot, not a live one-shot — same discipline the Sheet1 FR12 replay used.
