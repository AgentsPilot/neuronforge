## Why Single Stage After Grounding?

| Two-Stage (Old) | Single Stage After Grounding (New) |
|-----------------|-----------------------------------|
| Show clarifications after Phase 1 | Wait for grounding |
| Show confirmations after Phase 4 | Show everything in one UI |
| Questions are generic | Questions enriched with real data |
| "Which sheet?" | "Found 3 sheets: Sales, Q1, Old" |
| User interrupted twice | User answers once |
| Compile before asking | Ask before compiling |
| Wasted work if rejected | No wasted compilation |

**When:** After Phase 2 (Grounding) + 5-layer detection
**What:** ALL user decisions in one place:
- Questions (from `clarifications_needed[]`, enriched by grounding)
- Critical confirmations (from 5-layer detection)
- Assumptions (from `assumptions[]`, validated by grounding)
- Edge cases (from `edge_cases[]`)

**Purpose:** Single point of user interaction before compilation

|---------|-----------------|

| PII exposure risk | Layer 5: Business logic |

**UI Pattern:** Checkboxes, collapsed by default if validated

## Summary: Single-Stage Clarification System

| Section | Source | What It Catches |
|---------|--------|-----------------|
| **Questions** | `clarifications_needed[]` + grounding | LLM's known unknowns, enriched with real options |
| **Please Confirm** | 5-layer detection | LLM's unknown unknowns (patterns, risks, conflicts) |
| **Assumptions** | `assumptions[]` + grounding | Validated beliefs, with fake validations exposed |
| **Edge Cases** | `edge_cases[]` | What happens when things go wrong |

**Key Insight:** By waiting until after grounding, all user decisions happen in ONE place with REAL data options, not hypothetical ones.

The single UI ensures users approve:
1. All clarifications (with grounding-enriched options)
2. All pattern-detected issues (loop intent, PII risks)
3. All assumptions (validated or flagged)
4. All edge case handling

**Result:** Intent mismatches caught **before** IR formalization or compilation begins.
