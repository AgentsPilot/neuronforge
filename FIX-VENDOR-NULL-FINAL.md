# Final Solution for Vendor Null Issue

## Problem
The smart fallback detection updates step6's `description` field, but document-extractor doesn't use description during execution. It only looks at the `fields` parameter.

## Why Current Fix Doesn't Work
```typescript
// This updates description:
extractionStep.description += " If vendor not found, use Unknown Vendor";

// But document-extractor executor only looks at:
const { fields } = parameters;  // Description is not passed to extractor!
```

## Root Cause Analysis
1. `description` field is for **IntentContract generation** (design phase)
2. `fields` configuration is for **execution** (runtime)
3. Document-extractor doesn't have a way to specify fallback values in `fields` array
4. Variable resolution doesn't support `{{var || 'default'}}` syntax

## Proper Solutions (Choose One)

### Option 1: Add Transform Step (RECOMMENDED)
Add a step immediately after step6 to sanitize extracted fields:

```json
{
  "id": "step6b",
  "type": "transform",
  "operation": "map",
  "input": "{{extracted_fields}}",
  "config": {
    "type": "ensure_defaults",
    "defaults": {
      "vendor": "Unknown Vendor",
      "category": "Uncategorized",
      "type": "Unknown"
    }
  },
  "description": "Ensure required fields have default values"
}
```

**Pros:**
- ✅ Generic solution (works for any field)
- ✅ Doesn't modify extraction logic
- ✅ Easy to understand and maintain
- ✅ Can be auto-generated by calibration

**Cons:**
- Adds one extra step per scatter-gather loop

### Option 2: Enhance Variable Resolution
Add fallback syntax support to `ExecutionContext.resolveVariable()`:

```typescript
// Support: "{{extracted_fields.vendor || 'Unknown Vendor'}}"
const match = varRef.match(/\{\{(.+?)\s*\|\|\s*(.+?)\}\}/);
if (match) {
  const [, variable, fallback] = match;
  const value = this.resolveVariable(`{{${variable}}}`);
  return value || fallback.replace(/['"]/g, '');
}
```

**Pros:**
- ✅ Elegant syntax
- ✅ No extra steps needed
- ✅ Matches JavaScript semantics

**Cons:**
- Requires changes to core variable resolution
- Need to handle escaping and edge cases
- Might break existing workflows

### Option 3: Modify IR Compiler
Update `IntentToIRConverter` or `ExecutionGraphCompiler` to automatically insert validation/default steps when required fields come from nullable sources.

**Pros:**
- ✅ Fixes the root cause (per CLAUDE.md principles)
- ✅ Prevents the issue from happening
- ✅ No runtime overhead

**Cons:**
- Complex changes to compiler
- Needs semantic analysis of data flow
- Might be too aggressive
