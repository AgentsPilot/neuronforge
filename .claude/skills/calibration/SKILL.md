# calibration

Load this context **before** writing or changing code that touches the Calibration module — the post-creation process that takes a generated agent and makes it production-ready by running it, detecting issues, auto-repairing what it safely can, and guiding the user through the rest. It's the durable source of truth for what calibration does today, where each piece lives, and what must not be broken.

> **First read**: [`docs/Calibration/CALIBRATION_OVERVIEW.md`](../../../docs/Calibration/CALIBRATION_OVERVIEW.md) — the canonical index for the module. It links every authoritative doc by area (foundations, data model, detection layers, repair engines, hardcode sub-system, UX, operations) and carries the **Dead Code / Cleanup Backlog** (see § 9 here).

> **Where calibration sits**: an agent is generated by the **V6 pipeline** (see the `v6-pipeline` skill) → persisted by `/api/create-agent` → the user opens it in the **V2 Sandbox** (`/v2/sandbox/[agentId]`) and runs **calibration** to validate + repair it against real input before it goes to production. Calibration consumes the compiled DSL (`agents.pilot_steps` / `agents.workflow_steps`) and the V6 artifacts (`agent_config.ai_context.intent_contract`; the enhanced prompt lives in the `agents.user_prompt` column / `getAgentAiContextView` since the A2 de-dup — no longer in `ai_context`); it does not regenerate them.

## 1. Two surfaces — get this distinction right first

"Calibration" names **two cooperating mechanisms**. Most user requests are about surface (A). Know which one you're touching.

### (A) Post-creation Batch Calibration — user-triggered, the main surface

The user clicks "Run Calibration" in the Sandbox. The UI (`CalibrationSetup`) POSTs to **`/api/v2/calibrate/batch`** ([app/api/v2/calibrate/batch/route.ts](../../../app/api/v2/calibrate/batch/route.ts), ~4300 lines — the heart of the module). The route:

1. Authenticates, acquires a distributed lock (one calibration per agent at a time), opens a `calibration_sessions` row.
2. **Fast-path check** — computes `generateWorkflowHash` + `generateInputSchemaHash` ([lib/utils/workflowHash.ts](../../../lib/utils/workflowHash.ts)); if the hash matches a prior `calibration_history` row with `status='success'`, it runs a single verification instead of the full loop.
3. Otherwise enters the **calibration loop**: `while (loopIteration < MAX_ITERATIONS)` where `MAX_ITERATIONS = 10` (batch route ~L1018/L1202). Each iteration runs the three detection layers + structural/hardcode repair, applies auto-fix proposals silently, checkpoints state, and checks convergence.
4. On the iteration cap with issues still open, emits a **non-convergence signal** (`G-CAL-3`, batch route ~L3796) so the user isn't told "all good" when it isn't.
5. Persists the outcome to `calibration_history` (with `workflow_hash`) and updates `agents.*` (production-ready flag/status). Groups remaining issues via `IssueGrouper` for the wizard.

Workflow execution inside the loop is done by **`WorkflowPilot`** in batch-calibration mode (smart-continue: don't stop at first error, collect issues as `CollectedIssue[]`).

### (B) Runtime ShadowAgent — automatic, monitors live executions

[lib/pilot/shadow/ShadowAgent.ts](../../../lib/pilot/shadow/ShadowAgent.ts) is a **runtime health monitor**, invoked by `WorkflowPilot`/`ResumeOrchestrator` during *actual* agent runs while the agent is in the **`calibrating` lifecycle state** (before its first end-to-end success). It:

- Classifies step failures into 7 categories (`FailureClassifier`), stores **metadata-only** failure snapshots (NO client data).
- Manages the lifecycle: `calibrating → production → re-calibrating`. Active during calibration, **DORMANT** once `production_ready = true`, re-activates if a production agent fails.
- **Every ShadowAgent call is wrapped in try-catch — its failure must NEVER break the main execution flow.** Preserve this invariant in any change.

> The two share the `lib/pilot/shadow/` toolbox (detectors, `RepairEngine`) and the `calibrating`/`production_ready` lifecycle vocabulary, but they run in different places: (A) is a synchronous user-driven API loop; (B) is a passive runtime side-channel. When a request says "calibration isn't catching X" check (A) first; when it says "production agent got marked not-ready after a run" that's (B).

## 2. Architecture at a glance (surface A — the batch loop)

```
User clicks "Run Calibration" in /v2/sandbox/[agentId]   (CalibrationSetup.tsx)
        │  POST /api/v2/calibrate/batch
        ▼
Fast-path: workflowHash matches a prior status='success' history row?
   yes → single verification run, skip the loop            → CALIBRATION_STATUS_TRACKING
        │ no
        ▼
calibration_sessions row opened + distributed lock acquired
        ▼
LOOP  while (loopIteration < MAX_ITERATIONS=10):
   Checkpoint state (CheckpointManager)                     → CHECKPOINT_ROLLBACK
   Run workflow via WorkflowPilot (smart-continue)
   ── Layer 1  EnhancedSchemaValidator      (deterministic) → AUTO_REPAIR_CONFIG_AND_PARAMS / LAYER1_ROOT_LEVEL_ARRAY_PRIORITY
   ── Layer 2  ConstrainedSemanticValidator (LLM)           → LAYER2_NESTED_FIELD_DETECTION
                 ├ MultiStepStructuralDetector              → MULTI_STEP_STRUCTURAL_REPAIR / MISSING_FLATTEN_STEP_DETECTION
                 ├ ActionMismatchDetector                   → ACTION_MISMATCH_DETECTION
                 └ ScatterGatherFlowValidator               → SCATTER_GATHER_ERROR_PATTERNS
   ── Layer 3  DryRunValidator (real input)                 → LAYER3_DRY_RUN_IMPLEMENTATION
   Structural repair proposals (StructuralRepairEngine)     → STRUCTURAL_REPAIR_TRANSFORMATION_INTEGRATION
   Hardcode detection + parameterization (HardcodeDetector) → HARDCODE_REPAIR_SYSTEM / HARDCODE_DETECTION_V3_FINAL
   SmartLogicAnalyzer (cross-step logic)
   Apply confident auto-fixes silently; collect the rest
   Convergence check → break if converged / regressed (rollback)
        ▼
Iteration cap with open issues → non-convergence signal (G-CAL-3)
        ▼
IssueGrouper groups/prioritizes remaining issues for the wizard
        ▼
Story-driven wizard surfaces unfixed issues  (CalibrationStory)   → UX_REDESIGN_STORY_DRIVEN
        ▼
Outcome → calibration_history (+ workflow_hash) + agents.*         → CALIBRATION_HISTORY_IMPLEMENTATION
```

**Most calibration logic is deterministic.** Only Layer 2 (`ConstrainedSemanticValidator`) makes an LLM call. Bias fixes toward the deterministic layers and the prompt-side root cause (the V6 pipeline) rather than adding heuristics to the repair engines.

## 4. The three detection layers (surface A)

| Layer | Engine | Catches | Speed | Doc |
|---|---|---|---|---|
| **1** | `EnhancedSchemaValidator` | Invalid `{{config.X}}` refs, missing required params, root-level vs item-level array shape | ~100 ms, deterministic | `AUTO_REPAIR_CONFIG_AND_PARAMS`, `LAYER1_ROOT_LEVEL_ARRAY_PRIORITY` |
| **2** | `ConstrainedSemanticValidator` (+ `MultiStepStructuralDetector`, `ActionMismatchDetector`, `ScatterGatherFlowValidator`) | Fields at wrong nesting level, wrong plugin action for the params, missing intermediate steps, scatter-gather param mismatches | LLM call | `LAYER2_NESTED_FIELD_DETECTION`, `ACTION_MISMATCH_DETECTION`, `MULTI_STEP_STRUCTURAL_REPAIR_*`, `SCATTER_GATHER_ERROR_PATTERNS` |
| **3** | `DryRunValidator` | Runtime type mismatches + empty results that pass schema validation but fail on real data | executes the workflow | `LAYER3_DRY_RUN_IMPLEMENTATION` |

`MISSING_ACTION_AUTO_REPAIR` (steps missing the `action` field) can fire at any layer.

## 5. Repair, loop safety, and the hardcode sub-system

- **Auto-repair vs surface-to-user**: confident, deterministic fixes are applied silently inside the loop; everything else is collected, grouped (`IssueGrouper`), and shown in the wizard for user decision. New repair logic must be **plugin-agnostic** (see § 7) and prefer a clear surfaced issue over a risky silent rewrite.
- **CheckpointManager** snapshots agent state before each iteration; if issue count *rises* (regression), the loop rolls back. → `CHECKPOINT_ROLLBACK`.
- **Convergence detection** tracks which `(stepId, fixType)` pairs have been applied to prevent infinite fix-revert-refix cycles before `MAX_ITERATIONS=10`. Non-convergence at the cap raises the `G-CAL-3` signal. → `CONVERGENCE_DETECTION`.
- **Verdict + coverage floor (Item 6 / G1 anti-false-success).** The run's outcome is decided by `CalibrationVerdict.computeVerdict` (§ 3), not a raw issue count: a BLOCKING-class issue always fails; an unexercised or all-blank run resolves to `inconclusive` (never a clean pass); only allow-listed cosmetic user-confirm-only issues may be waved (Item 6a). The "exercised real path" signal comes from `dataQuality.deriveCoverageSignal` over the real per-step outputs. Preserve these guards — do not add a code path that lets a blocking/all-blank run reach `passed`.
- **Hardcode sub-system** runs inside calibration but is self-contained. The **V3 (current/authoritative)** approach reads `enhanced_prompt.specifics.resolved_user_inputs` from the V6 artifacts — it does NOT pattern-match or guess. This is the right model: the V6 pipeline already knows which literals came from user input, so calibration reads that ground truth instead of re-deriving it. → `HARDCODE_REPAIR_SYSTEM`, `HARDCODE_DETECTION_V3_FINAL`, `HARDCODE_REPAIR_FLOW`.

## 6. Data model

| Object | Type | Purpose |
|---|---|---|
| `calibration_sessions` | table | Active-run state; one open row per in-flight calibration (guarded by the distributed lock). Repo: `CalibrationSessionRepository`. |
| `calibration_history` | table | One row per completed calibration. Carries `workflow_hash` (fast-path), `status`, issue counts. Repo: `CalibrationHistoryRepository`. |
| `calibration_success_metrics` | **view** | Analytics over `calibration_history`. Empty until the migration is applied (gap noted in `CALIBRATION_PRODUCTION_READY_GAP_ANALYSIS`). |
| `agents.*` | columns | Calibration status / production-ready flag updated at the end of a run (see the two status columns below). |

**Verdict states (Item 6, carried in the API response + `calibration_history.metadata.verdict`).** `computeVerdict` returns one of `passed` / `failed` / `needs_review` / `inconclusive` / `corrected_not_verified`. The DB `calibration_history.status` CHECK allows only `success` / `failed` / `needs_review` / `verification_only`, so `inconclusive` + `corrected_not_verified` **map onto `needs_review`** at the DB layer (no migration); only a genuine `passed` writes `status='success'`. The plain-language labels live in `VERDICT_LABELS` (so "not tested" is never shown as "working").

**Two distinct `agents` status columns — don't conflate them:**
- `agents.last_calibration_status` (`success` / `needs_review` / `failed` / null) — the fast-path identity paired with `workflow_hash` (see `CALIBRATION_STATUS_TRACKING`).
- `agents.calibration_status` (`running` / `passed` / `failed` / `skipped`) — the wizard/gate column, set at the route tail via `AgentRepository.setCalibrationStatus`. It keys on `isCalibrationHistoryPass(latest.status)` (= history `status === 'success'`), so a **cosmetic-only pass now writes `'passed'`** (Item 6a) — previously it wrongly wrote `'failed'` because it also required zero remaining issues.

Known data gaps (tracked in `CALIBRATION_DATA_COMPLETENESS_ANALYSIS`): `v6_version`, `model_used`, `plugins_used`, `complexity_score` are not yet captured per run.

## 7. Platform principles calibration MUST follow

These are the project-wide rules from [CLAUDE.md § Platform Design Principles](../../../CLAUDE.md), restated because calibration is a frequent place they get violated.

| ❌ Wrong | ✅ Right |
|---|---|
| Add `"for Google Drive, use find_or_create_folder"` to a detector/repair engine | Reason from the plugin schema. Keep detectors plugin-agnostic; encode plugin specifics as `x-*` annotations on the plugin JSON, consumed generically. |
| Fix a generation bug by adding a repair in calibration | **Trace to the responsible V6 phase and fix it there.** Calibration repairs should only be generic compiler-style fixes that scale to any plugin (removing redundant steps, normalizing refs, inserting a flatten). Plugin-specific corrections belong upstream. |
| Hardcode field names / API patterns in a validator | Plugin schemas + `enhanced_prompt.resolved_user_inputs` are the source of truth. |
| Silently drop/short-circuit data because it looks empty | Surface it as an issue. Empty downstream data → fabricated content is the failure mode the V6 runtime guards (WP-10/WP-13). Calibration must not reintroduce it. |

**Root-cause rule:** when calibration *detects* a recurring class of issue, the fix usually belongs in the V6 pipeline (the `v6-pipeline` skill) so the agent is generated correctly next time — calibration's job is to *catch and safely repair*, not to become the place every plugin quirk gets hardcoded.

## 8. Decision guide — where to make different kinds of changes

| You want to… | Touch this | Don't touch |
|---|---|---|
| Add/adjust a deterministic check (config refs, params, array shape) | Layer 1 `EnhancedSchemaValidator` + its doc | The LLM layer — keep deterministic checks deterministic |
| Add a semantic check that needs reasoning over the whole workflow | Layer 2 `ConstrainedSemanticValidator` (structured-output constrained) | Don't bolt an LLM call onto Layer 1 or 3 |
| Detect a new structural gap (missing step/flatten) | `MultiStepStructuralDetector` (plugin-agnostic) | Don't hardcode a per-plugin step recipe |
| Detect wrong-action / wrong-param usage | `ActionMismatchDetector` driven by the plugin parameter schema | Don't add a `{action → action}` synonym map |
| Change what runs with real data | Layer 3 `DryRunValidator` | Don't move dry-run logic into the deterministic layers |
| Change the loop (iterations, convergence, rollback) | `batch/route.ts` loop (~L1018–L3820) + `CheckpointManager` + convergence tracking | Don't raise `MAX_ITERATIONS` to mask non-convergence — fix the fix-revert cycle |
| Change fast-path skip behaviour | `workflowHash.ts` + the fast-path block in `batch/route.ts` (~L138) + `CalibrationHistoryRepository` | Don't skip based on agent id alone — the hash is the safety |
| Change what the user sees | `CalibrationSetup` / `CalibrationStory` + `userFacing`/`friendlyLanguage` | The dead `CalibrationDashboard`/`WorkflowPreview` trees (§ 9) |
| Persist a new calibration metric | `CalibrationHistoryRepository` + a migration; populate it in `batch/route.ts` | Don't write directly to supabase from the route (repository pattern) |
| Change hardcode detection | `HardcodeDetector` reading `resolved_user_inputs` | Don't add regex/heuristic detection — V2 (heuristic) was archived for a reason |
| Fix "production agent got un-readied after a run" | Surface B: `ShadowAgent` + `FailureClassifier` + lifecycle flags | The batch route — that's surface A |

## 10. Out of scope (other skills / surfaces)

- **V6 agent-generation pipeline** (Phases 0–4: vocabulary → IntentContract → binding → IR → DSL). That's the `v6-pipeline` skill. Calibration *consumes* its output (`pilot_steps`, `enhanced_prompt`, `intent_contract`) and, per § 7, recurring detected issues usually get root-caused there.
- **The upstream chat creation flow** (`/v2/agents/new`, the Phase 1/2/3 conversation, the v16 prompt). That's the `agent-creation-flow` skill. It produces the Enhanced Prompt that V6 turns into the DSL that calibration validates.
- **The DSL runtime internals** (`StepExecutor`, `ParallelExecutor`, `ExecutionContext`, the V6 `StructuralRepairEngine` runtime path). Owned by `v6-pipeline`. Calibration drives `WorkflowPilot` but doesn't own step execution.
- **Plugin internals** (OAuth, connection UI, executors). The `new-plugin` skill. Calibration only reads plugin definitions to validate against them.

## 11. Anti-patterns to refuse

| ❌ Anti-pattern | ✅ Correct |
|---|---|
| "Add a per-plugin rule to a detector/repair engine so calibration fixes plugin X" | Plugin-agnostic detection + `x-*` annotations on the plugin JSON. Plugin-specific generation fixes go to V6 (§ 7). |
| "Raise `MAX_ITERATIONS` so it eventually converges" | Diagnose the fix-revert-refix cycle in convergence tracking. The cap + `G-CAL-3` exist to surface non-convergence, not hide it. |
| "Silently auto-rewrite the DSL when a ref looks wrong" | Prefer a surfaced issue with a clear before/after. Silent fuzzy rewrites corrupt valid workflows (the cautionary tale lives in the V6 `StructuralRepairEngine` history). Auto-fix only when schema-driven + high-confidence. |
| "Drop items that look empty so the run 'succeeds'" | Surface it. Empty data → fabricated output is the exact failure calibration exists to prevent. |
| "Bring back regex/heuristic hardcode detection" | V2 heuristic was archived; V3 reads `resolved_user_inputs` ground truth. |
| "Have ShadowAgent throw to stop a bad run" | ShadowAgent is wrapped in try-catch and must never break main execution. It observes and records; it doesn't gate. |
| "Call supabase directly from the batch route" | Repository pattern — `CalibrationHistoryRepository` / `CalibrationSessionRepository`. |
| "Add a new calibrate route by copying a deprecated stub" | Follow the canonical API pattern (`new-api-route` skill); build on `batch`. |
| "`console.log` to debug calibration" | Pino: `createLogger({ module: 'BatchCalibrationAPI', service: 'api' })`, child logger with correlation id. |

## 12. When you've finished work in this area

1. **`npx tsc --noEmit` clean** on touched files (the repo's `next.config.js` ignores TS errors — fix them anyway, per CLAUDE.md).
2. **Run the relevant tests**: `npx jest lib/pilot/shadow` for detector/repair work (note `__tests__/StructuralRepairEngine.wp32.test.ts`, `userFacing.test.ts`).
3. **Smoke-test a real calibration** end-to-end via `/v2/sandbox/[agentId]` → Run Calibration on an agent that exercises your change; watch logs with `monitor-calibration.sh` (`CALIBRATION_LOG_GUIDE`).
4. **Verify convergence + history**: confirm the loop converges (no `G-CAL-3`) and a `calibration_history` row is written with the right `workflow_hash` and `status`.
5. **Update the docs** under `docs/Calibration/`: if behaviour changed, update the relevant area doc and the **overview index** (Change History, NEWEST FIRST). If you removed dead code, update the **Dead Code / Cleanup Backlog** section.
6. **If the real fix was upstream** (a generation bug calibration was papering over), open/extend the matching V6 weak-point per the `v6-pipeline` skill instead of (or in addition to) the calibration repair.
