# CRITICAL: Intent Contract Schema Mismatch

**Date:** 2026-02-26
**Severity:** 🔴 BLOCKING - Pipeline cannot work with current mismatch
**Reported By:** OpenAI Review

---

## Executive Summary

The V6 pipeline has **two conflicting type definitions both named `IntentContractV1`**:

1. **Core DSL V1** ([core-vocabulary.ts](../lib/agentkit/v6/intent/core-vocabulary.ts)) - actively generated by Intent LLM
2. **Generic Intent V1** ([intent-schema-types.ts](../lib/agentkit/v6/semantic-plan/types/intent-schema-types.ts)) - expected by CapabilityBinder

**Result:** Phase 1 outputs JSON that Phase 2 (CapabilityBinder) cannot consume.

---

## The Two Conflicting Schemas

### Schema 1: Core DSL V1 (Currently Generated)

**Location:** `lib/agentkit/v6/intent/core-vocabulary.ts:308-322`

```typescript
export type IntentContractV1 = {
  version: "core_dsl_v1";
  goal: string;
  summary?: string;
  unit_of_work: "item" | "record" | "attachment" | "event" | "message" | "file" | "row" | "entity";
  plugins_involved: string[];
  questions?: { id: string; question: string; blocking: boolean; choices?: string[] }[];
  constraints?: { id: string; description: string }[];
  risks?: string[];
  steps: CoreStep[]; // ← KEY DIFFERENCE
};
```

**CoreStep characteristics:**
- Uses `type: "fetch" | "transform" | "loop" | "aggregate" | "deliver" | "ai_extract" | ...`
- Has `inputs: Record<string, { ref: string }>` with JSONPath refs like `$.stepId.outputKey`
- Has `outputs: Record<string, string>` with multiple output keys per step
- Uses `semantic_op` field for fetch/deliver (e.g., `"EMAIL.SEARCH"`)
- Uses `params?: Record<string, Value | Ref>` with concrete parameter names
- Expression system: `{ op: "gt", left: { ref: "$.stepId.field" }, right: 50 }`

**Used by:**
- ✅ Intent LLM generation ([generate-intent.ts](../lib/agentkit/v6/intent/generate-intent.ts))
- ✅ System prompt ([intent-system-prompt.ts](../lib/agentkit/v6/intent/intent-system-prompt.ts))
- ✅ Validation ([generate-intent.ts:46](../lib/agentkit/v6/intent/generate-intent.ts) checks `version === 'core_dsl_v1'`)

### Schema 2: Generic Intent V1 (Expected by Next Phase)

**Location:** `lib/agentkit/v6/semantic-plan/types/intent-schema-types.ts:202-227`

```typescript
export interface IntentContractV1 {
  version: IntentVersion; // "intent.v1"
  goal: string;
  unit_of_work?: UnitOfWorkSpec;
  config?: ConfigParam[];
  invariants?: Invariant[];
  required_outcomes?: string[];
  steps: IntentStep[]; // ← KEY DIFFERENCE
  confidence?: ConfidenceScore;
  meta?: JsonObject;
}
```

**IntentStep characteristics:**
- Uses `kind: "data_source" | "artifact" | "extract" | "loop" | "deliver" | "notify" | ...`
- Has `inputs?: RefName[]` - symbolic refs (just strings like `"emails"`)
- Has `output?: RefName` - single symbolic ref per step
- Uses `uses?: CapabilityUse[]` for semantic capabilities (domain + capability)
- Uses `CapabilityUse` with `{ capability: Capability; domain: Domain; hints?: string[] }`
- Expression system: `{ op: "test", left: ValueRef, comparator: Comparator, right?: ValueRef }`
- ValueRef system: `{ kind: "ref", ref: RefName, field?: string }`

**Used by:**
- ✅ CapabilityBinder ([CapabilityBinder.ts:3](../lib/agentkit/v6/capability-binding/CapabilityBinder.ts) imports from intent-schema-types)
- ⚠️ NOT used by Intent LLM generation

## Critical Mismatches (Why Pipeline Breaks)

### 1. Version String Mismatch
- **Core DSL:** `version: "core_dsl_v1"`
- **Generic Intent:** `version: "intent.v1"`
- **Impact:** Basic validation fails immediately

### 2. Step Structure Mismatch
- **Core DSL:** `steps: CoreStep[]` with `type` field
- **Generic Intent:** `steps: IntentStep[]` with `kind` field
- **Impact:** CapabilityBinder checks `step.kind` but receives `step.type`

### 3. Reference System Mismatch
- **Core DSL:** JSONPath refs like `{ ref: "$.stepId.outputKey" }`
- **Generic Intent:** Symbolic refs like `RefName = "emails"` + `ValueRef = { kind: "ref", ref: "emails", field?: "subject" }`
- **Impact:** Compiler cannot resolve data flow

### 4. Capability Binding Mismatch
- **Core DSL:** Uses `semantic_op: "EMAIL.SEARCH"` with concrete params
- **Generic Intent:** Uses `CapabilityUse = { domain: "email", capability: "search" }`
- **Impact:** CapabilityBinder cannot map semantic_op to plugin actions

### 5. Input/Output Schema Mismatch
- **Core DSL:**
  - `inputs: Record<string, { ref: string }>` (map of input aliases)
  - `outputs: Record<string, string>` (map of output descriptions)
- **Generic Intent:**
  - `inputs?: RefName[]` (array of symbolic refs)
  - `output?: RefName` (single symbolic ref)
- **Impact:** Data flow tracking completely incompatible

### 6. Expression System Mismatch
- **Core DSL:** `{ op: "gt", left: { ref: "$.stepId.field" }, right: 50 }`
- **Generic Intent:** `{ op: "test", left: ValueRef, comparator: "gt", right: ValueRef }`
- **Impact:** Conditional logic cannot be evaluated

## Root Cause Analysis

### Historical Context

Looking at the codebase structure, it appears:

1. **Generic Intent V1** was designed first as the "ideal" plugin-agnostic contract
2. **Core DSL V1** was created later for actual LLM generation (more concrete/executable)
3. Both are named `IntentContractV1` but serve different purposes
4. Pipeline phases were never updated to align on one schema

### Design Philosophy Conflict

**Generic Intent V1 philosophy:**
- Plugin-agnostic (no semantic_op, no concrete params)
- Execution-agnostic (symbolic refs, not JSONPath)
- Compiler binds capabilities → actions → IR

**Core DSL V1 philosophy:**
- Semi-bound (has semantic_op like "EMAIL.SEARCH")
- Execution-aware (JSONPath refs ready for IR compilation)
- Closer to executable IR format

## Solution Options

### Option A: Make Intent LLM Generate Generic Intent V1 (Recommended)

**Changes Required:**
1. Update `intent-system-prompt.ts` to teach Generic Intent V1 format
2. Update validation in `generate-intent.ts` to check `version === 'intent.v1'`
3. Update all examples/rules to use `kind` not `type`, symbolic refs, CapabilityUse

**Pros:**
- ✅ Aligns with original design philosophy (plugin-agnostic)
- ✅ CapabilityBinder works without changes
- ✅ Clean separation: Intent → Binding → IR → Compilation

**Cons:**
- ⚠️ Large prompt rewrite required
- ⚠️ More abstract (LLM must generate symbolic refs, not JSONPath)
- ⚠️ Need to regenerate all test contracts

### Option B: Update CapabilityBinder to Consume Core DSL V1

**Changes Required:**
1. Change CapabilityBinder imports to use CoreStep from core-vocabulary
2. Update binding logic to handle `type` not `kind`
3. Map `semantic_op` to plugin actions (different binding strategy)
4. Handle JSONPath refs instead of symbolic refs

**Pros:**
- ✅ Intent LLM prompt stays unchanged
- ✅ Generated contracts continue to work

**Cons:**
- ❌ Defeats plugin-agnostic design philosophy
- ❌ Binding becomes "mapping semantic_op to actions" not "capability → action"
- ❌ JSONPath refs bypass clean symbolic data flow
- ❌ Harder to maintain two parallel type systems

### Option C: Create Adapter/Translator Layer

**Changes Required:**
1. Create `CoreDSLToIntentAdapter` that translates Core DSL V1 → Generic Intent V1
2. Insert adapter between Intent generation and CapabilityBinder
3. Map `type` → `kind`, `semantic_op` → `CapabilityUse`, JSONPath → symbolic refs

**Pros:**
- ✅ Preserves both schemas
- ✅ Minimal changes to existing code

**Cons:**
- ❌ Adds complexity (translation layer)
- ❌ Translation may lose information or introduce errors
- ❌ Maintains two schemas indefinitely
